## 一、严重 Bug（必须修，影响功能或安全）

### CORE-01　失败计数器整段成为死代码
- 位置：`ChatViewModel.kt:130-132 / 240-252 / 335`
- 现象：循环顶部 `stepActionFailures.clear()` 在**每一轮 while 迭代**起始位置执行；而失败分支只是 `continue` 回到 while 起点，下一轮立刻被 clear 清零。结果 `stepFails >= MAX_STEP_ACTION_FAILURES (=2)` 永远为 false，242-245 行的"该动作连续失败，强制换策略"分支永不生效。
- 复现：让模型连续返回同一种 action 但每次失败（如越界 tap），观察 logcat —— 应触发硬拦截但实际仍重复执行直到 `retryCount > maxRetries(=4)` 才终止。
- 影响：硬拦截策略形同虚设；当模型陷入"同动作反复失败"时缺少早期阻断，多浪费数次模型请求与时间。

### CORE-02　extractActionType 对标准 JSON 永远返回 unknown
- 位置：`ChatViewModel.kt:436-460`
- 现象：正则只识别 `action="..."`（等号分隔），但模型按规范输出的 JSON 是 `"action": "Tap"`（冒号分隔），两个匹配模式都不命中 → 返回 `"unknown"`。
- 影响链：
  1. `task_steps.action_type` 字段所有 JSON 模式响应都被存为 `"unknown"`，历史详情页"操作类型"列展示无意义；
  2. CORE-01 的 `stepActionFailures` 以 actionType 为 key，所有失败都落入同一个 `unknown` 桶（即便 CORE-01 修了也不准）；
  3. `if (actionType == "type" && ... currentApp == "com.tencent.mm")` 判定失效，微信里 `type` 失败时**走不到**专门的输入法切换提示路径。

### CORE-04　finish 检测会被字符串内容误触发
- 位置：`ChatViewModel.kt:232-234`
- 现象：判定 `response.action.lowercase().contains("finish(")`，未加边界。若模型返回 `do(action="Type", text="say finish( hello")` 这类把字面量 `finish(` 写入 text 参数的 action，`contains` 立即命中，被错判为任务完成。
- 影响：未真正完成的任务被报告为成功。结合 `result.message` 来自 `processActionObject`，可能带上"输入成功"等字样进一步误导用户。

### CORE-05　非取消类 HTTP 异常导致 DB 会话永久卡 running
- 位置：`ChatViewModel.kt:198-208` + `MainActivity.kt:425-441`
- 现象：`while` 内的 `try/catch` 仅当 `_cancelRequested.value == true` 时才认作取消并干净返回，其余异常直接 `throw e` 抛出，循环外**没有兜底 try**。最终被 `MainActivity.executeTask` 的 lifecycleScope catch，仅 `addSystemMessage("执行失败：...")`，**永远不会**调用 `db.taskSessionDao().updateStatus(sessionId, "fail")`。
- 影响：任何 5xx / JSON 解析失败 / 连接重置都会让 `task_sessions` 行的 `status="running"` 永久滞留，HistoryActivity 列表错误地显示这条任务"进行中"。`MyAccessibilityService` 重启或 app 杀进程也无清理机制。

### CORE-08　两条入口可同时启动两个 executeTaskLoop
- 位置：`MainActivity.kt:425-441` + `FloatingWindowManager.kt:411-423`
- 现象：`MainActivity.executeTask` 与 `FloatingWindowManager.onVoiceResult` 都直接 `viewModel.executeTaskLoop(...)`，没有任何并发护栏。任务执行中再次发送/语音 → 启动第二个循环；两条循环共享 `companion object _executionState`、共享 `messageContext`，互相覆盖。
- 复现：在悬浮窗任务执行中，前台打开主页快速点击发送。
- 影响：状态错乱；`messageContext` 是非线程安全 `mutableListOf`，并发 add/remove 可能抛 `ConcurrentModificationException`；`ModelClient.currentCall` 是单字段，第二条会覆盖第一条的 Call 引用，第一条的 HTTP 请求无法被 cancel。

### CORE-11　tap 不校验坐标范围
- 位置：`ActionExecutor.kt:432-483`
- 现象：`swipe`(608)、`longPress`(745)、`doubleTap`(813) 都对 0..1000 做了校验；唯独 `tap` 直接拿 `arr[0].asFloat / arr[1].asFloat` 进 `relativeToAbsolute`。模型若返回 `[1500, 800]` 或负值，转出屏幕外坐标，`dispatchGesture` 报 `onCancelled`，最终给模型反馈的失败原因只是"点击手势提交失败"，模型无法判断是越界还是被遮挡。
- 影响：失败提示模糊，模型重试时仍可能继续给越界坐标，浪费多次重试。

### NET-01　Release 包仍打印完整请求体（含 API Key 与截图 base64）
- 位置：`ModelClient.kt:50-53`
- 现象：`HttpLoggingInterceptor` 的 level 写死为 `BODY`，没有 `BuildConfig.DEBUG` 守门。release 安装后 logcat 持续输出 `Authorization: Bearer <真实 Key>` + 完整 base64 截图（用户屏幕原图）+ 模型完整响应。
- 影响：任何能读 logcat 的途径（同 UID 应用、root 设备、电脑端 `adb logcat`）都能拿到智谱 API Key 与历史截图。

### NET-04　非 IOException 网络错误绕过取消通道
- 位置：`ModelClient.kt:117-132` + `ChatViewModel.kt:198-208`
- 现象：5xx 在 `onResponse` 中以 `Exception("请求失败: ...")` 抛出（**不是** `IOException`）。`ChatViewModel` 只在 `_cancelRequested` 时才把异常吞下；其他情况 throw 出去，配合 CORE-05 触发"孤儿 running 会话"。
- 影响：网络瞬断或服务端 5xx 都会污染历史库。

### NET-11　导出配置文件以明文 JSON 写入用户选择路径
- 位置：`ApiConfigBackup.kt:41-53`
- 现象：通过 SAF 打开 `CreateDocument`，把 `zhipu_api_key`、`baidu_api_key`、`baidu_secret_key` 三个明文字段直接 `gson.toJson(...)` 写入 uri。无加密、无密码、无导出告警。
- 影响：导出位置很可能是 Download 目录或同步至云盘的路径，文件一旦被备份/分享/被其他应用扫描，三个密钥即外泄。

### FLOAT-01　悬浮窗每次语音命令都新建 ChatViewModel（永久泄漏）
- 位置：`FloatingWindowManager.kt:411-423`
- 现象：`onVoiceResult` 内 `val viewModel = ChatViewModel(service.application)`，绕过 `ViewModelProvider`，永远不会触发 `onCleared()`；`viewModelScope` 永不取消，`init` 块中的 `viewModelScope.launch { ... }` 也长驻。
- 影响：每次唤起一次语音命令泄漏一个 `ChatViewModel` + 一份 `messageContext` 内存 + 一组 StateFlow 订阅；长时间使用悬浮窗后内存与协程数会显著膨胀。

### UI-01　主界面每次发送任务都新建 ChatViewModel
- 位置：`MainActivity.kt:425-441`
- 现象：同 FLOAT-01。`executeTask` 中 `chatViewModel = ChatViewModel(application)` 反复覆盖字段；旧实例的 `viewModelScope` 仍持有 `application` 引用直到自然 GC。
- 影响：与 FLOAT-01 同性质的泄漏；多次发送后 `ChatViewModel.companion object` 上的 StateFlow 被多个不同实例并发读写。

### MANIFEST-06　allowBackup=true + 默认空备份规则
- 位置：`AndroidManifest.xml:36-44` + `res/xml/backup_rules.xml` / `data_extraction_rules.xml`
- 现象：`android:allowBackup="true"`，但两份备份规则文件都是默认空模板（无 include/exclude 声明）。系统按"全量备份"处理 → DataStore（含智谱/百度 API Key）+ Room DB（任务历史 + AI thinking + 步骤）随设备备份上传 Google 云。
- 影响：用户只要开启了系统备份，三组密钥与历史命令上云；攻击面扩大到云端。

### BUILD-01　Release 没有启用 R8/混淆
- 位置：`build.gradle.kts:23-31`
- 现象：`release { isMinifyEnabled = false }`。`proguard-rules.pro` 即使存在也不参与编译。
- 影响：（1）Smali 反编译可直接读到所有类名/方法名（如 `ModelClient`, `extractActionType` 等内部逻辑）；（2）prompt 模板、模型名、AutoGLM 端点、百度端点 URL 全部明文留存于 APK；（3）APK 体积偏大；（4）第三方库无法被消减。

## 二、中等严重度问题

### CORE-03　失败重试时重复添加完整原始用户消息
- 位置：`ChatViewModel.kt:172-194` + `354`
- 现象：`stepCount` 仅在成功后递增，失败 `continue` 时仍为 0；下一轮 `if (stepCount == 0)` 又走 177 行 `client.createUserMessage(userPrompt, screenShot, ...)`，把完整 prompt + 新截图再 push 一次。
- 缓和：220-228 行的 `removeImagesFromMessage` 会在每轮 assistant 写入后剥离 `messageContext.size-2` 处的图片，所以**图片膨胀**被控制；但**完整文本 prompt** 在每次失败后仍累积一份。
- 影响：4 次重试可累积 4 份原 prompt 文本 + 4 条 errortext，token 数无谓增长，长 prompt 模板下尤其明显。

### CORE-06　actionExecutor 初始化竞态
- 位置：`ChatViewModel.kt:82-88` + `247-251`
- 现象：`init { viewModelScope.launch { ... actionExecutor = ActionExecutor(service) } }` 异步初始化。若用户进入 MainActivity 后立刻发送任务，`executeTaskLoop` 在 init 协程跑完前就调到 `actionExecutor?.execute(...)` → 返回 null → 250 行 elvis 兜底 `ActionResult(false, "ActionExecutor is null")` → 触发首次失败。
- 影响：首次发送任务在冷启动场景下大概率第一步白白失败一次，重试时通常已自愈，但日志里会留下莫名其妙的 "ActionExecutor is null"。

### CORE-07　messageContext 非线程安全
- 位置：`ChatViewModel.kt:34, 92, 217-228, 313-316`
- 现象：`mutableListOf<ChatMessage>()` 无任何 `synchronized` / `Mutex` 保护。当 CORE-08 触发并发任务时，两条 add/remove/clear 互踩可能抛 `ConcurrentModificationException`，或顺序错乱导致模型读到错位的对话。
- 关联：CORE-08 是触发条件，并发不发生时此问题是潜伏风险。

### CORE-09　companion object StateFlow 跨 ViewModel 实例污染
- 位置：`ChatViewModel.kt:49-79`
- 现象：`_executionState` / `_cancelRequested` 是 companion object 上的进程级单例。MainActivity 与 FloatingWindowManager 各自 `new ChatViewModel(application)`（参见 FLOAT-01 / UI-01），都向同一份 StateFlow 写。配合 CORE-05 的孤儿异常路径，`_executionState.isRunning` 可能再也回不到 false。
- 影响：下次 MainActivity 启动时 `observeExecutionState` 看到 `state.isRunning && taskTitle != lastTaskTitle` → 又重放一遍"正在执行：旧任务"。

### CORE-12　wait 数字单位歧义（数字 vs 字符串行为不一致）
- 位置：`ActionExecutor.kt:859-883`
- 现象：
  - 纯 JSON 数字 `delay: 500` → `prim.asDouble.toLong() * 1000` → 当**秒**处理（实际 500s，被 30s 上限截断）
  - 字符串 `"500"` → 873-877 当**毫秒**处理（500ms）
  - 字符串 `"500 seconds"` → 当秒处理
- 影响：模型按"毫秒"思维返回纯数字时，会被错误放大 1000 倍，最终被 30s 上限截断，wait 总是停 30 秒。

### CORE-15　launchApp 在 Android 10+ 受 BAL 限制
- 位置：`ActionExecutor.kt:402-413`
- 现象：从 `AccessibilityService` 上下文调 `service.startActivity(intent)`。Android 10+ 起，应用在后台时 BAL（Background Activity Launch）会拦截 `startActivity`；`FLAG_ACTIVITY_NEW_TASK` 不能绕过。AccessibilityService 在 framework 层有部分豁免，但厂商深度系统（小米/华为）裁剪不一致，常见静默失败。
- 影响：从悬浮窗触发 `Launch app=微信` 在某些设备上看起来"啥也没发生"，被 422 行 catch 转为"启动 xxx 失败：..."。

### CORE-16　对外层就是纯 JSON 的响应可能误走"修复"分支
- 位置：`ActionExecutor.kt:201-229` + `extractJsonFromText:279-321`
- 现象：`extractJsonFromText` 对纯 JSON 输入直接返回 `trimmed`；但 `execute()` 接着判定 `if (jsonString.isEmpty() || jsonString == actionJson.trim())` —— 第二个条件在"输入本身就是 JSON" 时为 true，直接走 `tryFixMalformedJson` 路径；该函数对纯 JSON 不匹配任何规则，返回空，最终回退为"无法从响应中提取..."。
- 实际触发条件：`ModelClient.parseResponse` 在没有 `<answer>` 包裹但 action 已经是 `{...}` 的情况下命中。208-218 行尝试做了一次提取，多数时候能避开此分支；但若模型直接返回纯 JSON 而前面也没 thinking 内容，就会撞上这条路径。
- 影响：偶发性"无法从响应中提取有效的 JSON 动作"，模型能解析的输出被代码自己拒掉。

### NET-02　ModelClient.currentCall 是单字段（与 CORE-08 联动）
- 位置：`ModelClient.kt:44, 115-116, 175-178`
- 现象：`currentCall` 没有锁、没有 List；并发请求时后调覆盖前调，`cancelCurrentRequest()` 只能 cancel 最近一个 Call。

### NET-03　多 `<answer>` 块时 action 被拼坏
- 位置：`ModelClient.kt:191-194`
- 现象：`split("<answer>", limit = 2)` 后 `parts[1].replace("</answer>", "")` 全局替换。若模型输出 `<answer>A</answer><answer>B</answer>`（罕见但不是不可能），最终 `action = "A\<answer>B"` —— 中间残留的 `<answer>` 没被剔除。
- 影响：罕见模型抖动场景下 action 字符串损坏，下一轮解析失败。

### NET-12　ApiConfig 不做 Key 长度/字符校验
- 位置：`ApiConfigActivity.kt`（保存逻辑）+ `SettingsPrefs.kt:79-87`
- 现象：仅对输入做 `.trim()`，不过滤中间换行 `\n` / 制表符等。用户粘贴含尾部换行的 Key 时，DataStore 把换行也存下来；`Bearer $apiKey` 拼出的 header 含换行 → OkHttp 抛 `Unexpected char ... in header value`。
- 影响：用户复制 Key 时一不小心带回车就配置失败，错误信息晦涩。

### NET-13　import 不校验字段是否为空，会用空值覆盖现有配置
- 位置：`ApiConfigBackup.kt:58-78`
- 现象：DTO 反序列化后只校验 `version != null && version == CURRENT_VERSION`，不校验三个 key 字段是否为空。一个 `{"version":1, "zhipu_api_key":"", "baidu_api_key":"", "baidu_secret_key":""}` 的文件能被合法 import → 现有配置被空值覆盖。
- 影响：用户误导入一份残缺/老空配置，本地有效 Key 直接没了，且无 undo。

### SPEECH-02　百度 access_token 永不刷新
- 位置：`BaiduSpeechManager.kt:199-208 / 390-410`
- 现象：`accessToken` 字段一次获取后只在 destroy 重建实例时才失效。百度 access_token 默认 30 天有效，长期运行的应用会带过期 token 调用 ASR → 401 → recognize 返回 null → UI 仅显示"未识别到语音"，实际是 token 过期。无 401 自动重取逻辑。
- 影响：长期不重启 app 的用户某天会突然全部识别失败且原因不明。

### SPEECH-04　audioRecord.startRecording() 返回值未检查
- 位置：`BaiduSpeechManager.kt:262-263`
- 现象：调用后未读取 `recordingState`。某些设备处于勿扰、屏幕共享、第三方录音占用 mic 的情况下，`startRecording` 会静默失败但不抛异常，state 维持 STOPPED → 后续 `read` 返回 -1 → 空音频结束 → "未识别到语音"。
- 影响：失败原因模糊。

### SPEECH-05/06　cancel/destroy 无法中断同步 OkHttp 请求
- 位置：`BaiduSpeechManager.kt:115-128 / 415-466`
- 现象：`recognize` 用 `client.newCall(request).execute()`（同步阻塞）。协程被 cancel 时不会传播到 `execute()`；用户取消后音频已经上传完，只是结果被 sessionCounter 守门后丢弃。
- 影响：浪费一次完整上传带宽（音频可能数 MB），并继续占用 IO 线程到响应回来。

### VOSK-01　模型复制中断恢复缺失
- 位置：`VoskRecognizerManager.kt:64-148`（基于上一轮报告与代码结构）
- 现象：`copyDirectory` 中途崩溃或被杀进程会留下半成品目录；下次 `loadModel` 看到 `modelDir.exists()` 即跳过复制 → 直接加载半成品 → 永久不可用，必须卸载重装。无 `.complete` 标记或 checksum 校验。
- 影响：低概率但一旦发生用户无法自愈。

### VOSK-05　唤醒词到百度命令识别的麦克风交接竞态
- 位置：`WakeWordListeningManager.kt:onWakeWordDetected` + `FloatingWindowManager.kt:146-154`
- 现象：唤醒词命中后 `handler.post { listener?.onWakeWordDetected() }` 异步派发；`FloatingWindowManager` 立即 `stopWakeWordListening()` + `speechManager.start()`。Vosk 这一侧的 `break@loop` 与 `stopRecording()` 仍在跑，两侧短暂同时持有 mic 句柄 → 第二个 `AudioRecord` 初始化可能 `state != INITIALIZED` → 百度报"录音初始化失败"。
- 影响：偶发的"唤醒成功但接下来命令录音失败"。

### SVC-04　无障碍服务 destroy 与 floatingWindowManager 的访问竞态
- 位置：`MyAccessibilityService.kt:71-84, 109-114` + `FloatingWindowManager` 访问点
- 现象：`floatingWindowManager` 是 `var (private set)`。SettingsActivity / MainActivity 通过 `getInstance()?.floatingWindowManager?.xxx()` 调用。在系统标记 service 即将销毁但 `onDestroy` 尚未跑完的窗口期，调用 `WindowManager.addView` 可能抛 `BadTokenException` / `IllegalStateException`。
- 影响：低概率但日志会出现奇怪的 WindowManager 异常。

### INPUT-02　剪贴板兜底覆盖用户原内容
- 位置：`AccessibilityTextInput.kt:56-87 tryClipboardPaste`
- 现象：先 `setPrimaryClip(text)`、ACTION_PASTE、再 `setPrimaryClip("")`。流程中：（1）写入到 paste 之间，其他注册了 `OnPrimaryClipChangedListener` 的应用（特别是常驻输入法）有机会读取敏感文本；（2）末尾的 `""` 把用户原本的剪贴板内容也清空，不可恢复。
- 影响：隐私 + 体验。从模型让 AI 输入密码、验证码等场景尤其敏感。

### FLOAT-02　执行卡片销毁链双轨控制
- 位置：`FloatingWindowManager.kt:482-498` + `ExecutionCardView.kt:87-118`
- 现象：
  - `ExecutionCardView.showCompletion` 内部 `postDelayed` 5000ms 自销毁
  - `FloatingWindowManager` 的 `postTaskCleanup` 5500ms 后置字段为 null
  - 时序正常时：5000ms 自销毁先发生 → 5500ms cleanup 把已 null 的字段再 null 一次（OK）
  - 时序异常时（5000ms 自销毁 `removeView` 抛异常被吞），5500ms cleanup 执行 `executionCardView = null`，但 view 仍挂在 WindowManager → 永久残留卡片 + 无引用销毁。
- 影响：极端时机下屏幕残留无主卡片。

### FLOAT-08　拖拽边界用 Display.mode.physicalWidth/Height
- 位置：`CircleFloatingView.kt:73-86`
- 现象：使用物理像素而非可用区域。横屏/分屏/折叠屏（特别是折叠态）会让圆圈被吸附到屏幕外。`runCatching` 兜底用 `displayMetrics.widthPixels` 是正确的，但仅在异常时触发。
- 影响：折叠屏与分屏多任务场景下圆圈消失，需用户重启服务。

### FLOAT-12　hideForScreenshot/restore 期间布局变更引发异常
- 位置：`FloatingWindowManager.kt:321-368`
- 现象：截屏前 `removeView` → 截屏中用户旋转屏幕或锁屏 → `restoreAfterScreenshot` 用旧 layoutParams 调 `addView` → `BadTokenException` 或 display 失效。代码 try/catch 已吞掉，但卡片状态变为隐藏。
- 影响：视觉上"卡片消失"，下一轮 step 才能恢复。

### DATA-01　Room schema 没有 Migration 也没有 destructive 兜底
- 位置：`AppDatabase.kt:12-16`
- 现象：`version = 1` + `exportSchema = false`，未声明 `Migration`，未调 `fallbackToDestructiveMigration()`。当前没有立即风险（首版），但产品迭代任何字段变更都会让升级用户直接 `IllegalStateException` 闪退。
- 影响：未来升级隐患，建议在下个版本引入前先解决。

### A11Y-04　canRequestFilterKeyEvents=true 但代码无 onKeyEvent 实现
- 位置：`res/xml/accessibility_service_config.xml:13`
- 现象：声明了过滤按键事件能力（键盘记录器级别权限），但代码并未重写 `onKeyEvent`。
- 影响：上架审核会被标记为高风险；权限完全闲置。

### MANIFEST-01　QUERY_ALL_PACKAGES 触发 Play 政策审查
- 位置：`AndroidManifest.xml:7-8`
- 现象：实际 `AppRegister` 通过 `getInstalledApplications` 列举所有 app，是必需的。但 Play Store 对该权限有特殊政策，需提交豁免申请。
- 影响：侧载安装无问题；想上 Play 必须提交说明或改用 `<queries>` 限定（已部分声明 LAUNCHER 但仍保留 QUERY_ALL_PACKAGES）。

### MANIFEST-02 / 03　两枚 system-only 权限是死权限
- 位置：`AndroidManifest.xml:18-19, 23-26`
- 现象：
  - `MOUNT_UNMOUNT_FILESYSTEMS`：system-only，普通应用申请无效，代码无任何挂载操作。
  - `WRITE_SECURE_SETTINGS`：需 adb 授权或系统签名才有效。注释提到"无感重启无障碍服务"，但代码实际未用此 API。
- 影响：上架时会被审核质疑；占据权限清单显得激进。

### MANIFEST-04　MANAGE_EXTERNAL_STORAGE 与代码不匹配
- 位置：`AndroidManifest.xml:20-21`
- 现象：代码中 DataStore / Room 都在私有目录；`ApiConfigBackup` 走 SAF Uri（不需要此权限）。该权限完全闲置。
- 影响：与 MANIFEST-01 同样存在 Play 政策门槛。

## 三、低严重度 / 健壮性问题

### CORE-13　`tryFixMalformedJson` 中无引号参数正则不支持 Unicode
- 位置：`ActionExecutor.kt:127`
- 正则 `[A-Za-z_][A-Za-z0-9_]*` 不匹配 CJK。模型若用 `do(action=Tap, app=微信)` 这种无引号 + 中文写法，`app` 参数捕获失败。
- 触发概率低（模型一般会加引号），属于隐藏脆弱点。

### CORE-14　`asFloat` 抛 NumberFormatException 时反馈过于笼统
- 位置：`ActionExecutor.kt:436-444 / 547-559`
- 模型若返回 `element: ["10%", "20%"]` 或含 null，`asFloat` 抛异常被外层 `catch (e: Exception)` 接住，反馈仅"解析动作失败：..."，没有指明 element 字段类型问题。模型重试时缺乏方向。

### NET-05　bitmapToBase64 后软拷贝 bitmap 不 recycle
- 位置：`ModelClient.kt:395-401` + `MyAccessibilityService.kt:319-343`
- 上一轮报告把"硬件 bitmap 泄漏"严重度调得太高，事实上 `hardwareBuffer.close()` 已正确关闭原生缓冲；剩下的是 JVM 堆里的 `Bitmap` 对象与软拷贝的 ARGB_8888 内存（~8MB/张），等 GC 回收。长会话（10+ 步）短期内堆内存占用上升明显，但不至于 OOM。
- 影响：内存峰值偏高，建议在 base64 完成后立即 `.recycle()` 软拷贝。

### NET-07　无 TLS 版本限制 / 证书 pinning
- 旧设备可能允许 TLS 1.0；MITM 攻击下 API Key 与截图明文。属于通用风险，非项目特有。

### NET-09 / NET-10　Retrofit DTO 与 AutoGLMApi 是死代码
- `ChatResponse.kt`、`AutoGLMApi.kt` 在仓库内引用为 0，但容易让新人误以为是网络入口。建议明确删除或加注释。

### SVC-05　每次手势前后各 50ms 的芯片摘装延迟
- 位置：`MyAccessibilityService.kt:252-273`
- 50 步任务额外消耗 ~5s。属于已知 trade-off（防芯片吃手势），暂不动业务。

### SVC-03　API < 30 截图能力降级
- `getScreenshot` 在 API < 30 直接 `callback(null)`，正确。配合 `ChatViewModel.kt:161-170` 的 isMyProjectApp 区分，行为符合预期。仅作能力提示。

### FLOAT-04　悬浮窗任务异常时调 ChatViewModel.resetState() 影响主界面
- 位置：`FloatingWindowManager.kt:418-422`
- companion 静态方法直接重置全局 `_executionState`。极少数情况：用户在悬浮窗任务异常时主界面正在跑别的任务，全局状态被一并重置。

### FLOAT-10 / 11　ExecutionCardView / ExecutionCancelChipView 销毁路径异常时旧 view 残留
- 位置：`ExecutionCardView.kt:87-98`、`ExecutionCancelChipView.kt:91-102`
- `removeView` 抛异常被吞 → `rootView = null` → 下次 `create` 重新 inflate → 旧 view 仍挂着。WindowManager 视图泄漏。

### DATA-02　status 字段是 String，无 CHECK 约束

### DATA-04　HistoryActivity 全量 `getAll()` 无分页

## 五、设计建议（不修改原有业务流程，仅作提醒）

> 以下仅为风险提示与可选改进方向，**不要求改动**。是否调整由产品/作者自行判断。

1. **任务并发互斥**：MainActivity 与 FloatingWindowManager 任意一边任务运行时禁用另一入口（解决 CORE-08 / EDGE-05）。
2. **统一 ChatViewModel 单例**：当前两条入口各自 `new`，泄漏 + 状态分裂。可改用 `ViewModelProvider` 配合 Application-scoped factory，或抽出"任务执行控制器"为单例，ViewModel 仅做 UI 桥接。
3. **回放 / 重启动孤儿会话**：app 启动时扫描 `status='running'` 的旧会话，统一改为 `'fail'` 或新增 `'interrupted'` 状态。
4. **action 解析格式收敛**：`strings.xml` 系统提示词列出的 action 名比 `ActionExecutor` 实际实现的多（含 `Type_Name / Interact / Note / Call_API / Take_over` 等未实现项）。要么把 prompt 收窄到实际支持的 10 种，要么补齐执行器，避免模型偶尔返回未实现的 action 静默失败。
5. **JSON 请求 vs do() 函数式统一**：`parseResponse` 与 `parseAction` 都做了双格式兜底，但 `extractActionType`（CORE-02）只识别函数式。建议挑一种为主，另一种为兜底。
6. **API Key 加密**：`EncryptedSharedPreferences` 或 Android Keystore wrap，防 root 设备 + 防全量备份外泄（同时收紧 backup_rules）。
7. **导出文件加密 / 警告**：见 NET-11。最简方案：导出前提示"文件含明文密钥"，或要求输入密码加 zip。
8. **Manifest 权限瘦身**：删除 MOUNT_UNMOUNT_FILESYSTEMS / WRITE_SECURE_SETTINGS（实际不可用）；MANAGE_EXTERNAL_STORAGE 评估必要性；A11Y `canRequestFilterKeyEvents` / `canRequestTouchExplorationMode` 关掉；A11Y eventTypes 改 `typeWindowStateChanged`。
9. **Release 启用 R8 + ProGuard**：BUILD-01。
10. **HttpLoggingInterceptor 用 BuildConfig.DEBUG 守门**：NET-01。
11. **手势失败时反馈"为什么"给模型**：CORE-11 / CORE-14。当前只反馈"点击手势提交失败"。建议把校验失败、`onCancelled`、坐标越界这三类区分写入 `ActionResult.message`。
12. **token 过期自动重取**：SPEECH-02。捕获 401 / err_no=110 之类错误码后清掉 `accessToken` 重试一次。
13. **Vosk 模型完整性校验**：VOSK-01。复制后写一个 `.complete` 标记，loadModel 校验存在；缺失则重新复制。
14. **LRU 限制 messageContext**：长会话累积无限增长。建议保留最近 N 条；token 超限时滑窗。
15. **悬浮窗执行卡片销毁链单一来源**：FLOAT-02。
16. **历史 status 改 enum**：避免 DATA-02 拼写错风险。

## 九、二次复核修订（2026-05-03 当日追加）

> 本节是对前八节的内部复核结果。在提交报告后，我重新对照 `strings.xml` 中的 `system_prompt_template` 与 `res/xml/backup_rules.xml`、`res/xml/data_extraction_rules.xml` 三处事实源做了二次核对，发现前面有几条判断**不成立**或**严重度被高估**，需要在此明确修订。规则：保留原编号便于回溯，**修订后的结论以本节为准**。

### 9.1 应剔除的误报

#### ❌ CORE-02（之前列为高严重度）—— 误报，剔除

- **原结论**：`extractActionType` 对标准 JSON 响应永远返回 `unknown`，导致 DB `actionType` 字段都是 unknown、微信 type 失败专用提示路径失效等连锁问题。
- **复核事实**：`res/values/strings.xml` 的 `system_prompt_template` 明确要求模型输出**函数式**格式：
  ```
  - do(action="Tap", element=[x,y])
  - finish(message="xxx")
  ```
  而不是 JSON。`ModelClient.parseResponse` 也是按函数式提取（line 195-204 兜底 `finish(message=` / `do(action=`）。`response.action` 进入 `extractActionType` 时形如 `do(action="Tap", element=[500,600])`。
- **再验证 extractActionType**：
  - 正则 `action\s*=\s*["']([^"']+)["']` 命中 `action="Tap"` → 返回 `"tap"` ✓

- **唯一残留风险**：模型违反 prompt 输出 JSON 时（极罕见），DB 的 actionType 字段会是 `unknown`，但功能不受影响。仅作 INFO 级备注。

### 9.2 严重度需要降级的条目

#### ⬇️ CORE-04　高 → 低
- **原结论**：finish 检测被 `contains("finish(")` 字符串匹配误触发。
- **复核事实**：第三个条件 `response.action.lowercase().contains("finish(")` 是为了正常命中 system prompt 要求的 `finish(message="xxx")` 函数式输出，**这是设计的正常路径**而非 bug。误触发场景只在 type 文本本身包含 `finish(` 字面量时才发生（如让 AI 翻译/输入英文代码片段）。

#### ⬇️ FLOAT-01 / UI-01　高 → 中（同时修正描述）

  - `viewModelScope` 是 `ViewModel` 实例的成员字段，不是进程级单例；ChatViewModel 实例无强引用后可被 GC，viewModelScope 跟随实例回收。
  - `init { viewModelScope.launch { ... } }` 协程执行完即退出，不阻止 GC。
  - `executeTaskLoop` 期间持有引用，结束后局部变量 `viewModel` / `chatViewModel` 释放，可被 GC。
  - 所以不存在"永久泄漏"。
- **真正的问题**：
  1. **重复创建开销**：每次任务都新建 `OkHttpClient`（在 `ModelClient.init` 中）+ `ChatViewModel` 实例 + 对 `BaiduSpeechConfig.credentialsFlow` 等的订阅，浪费资源。
  2. **多实例并发污染 companion StateFlow**：CORE-08 触发时，多个 ChatViewModel 实例同时向 `_executionState` 写入，状态错乱。
  3. **短期内存峰值**：旧实例的 `messageContext`（含截图 base64）在被 GC 前与新实例并存。
- **修订严重度**：高 → 中（不再描述为"永久泄漏"）。

## 十、三次复核修订（2026-05-03 当日追加）

### 10.1 撤回：CORE-02 不应完全剔除，应改为"降级到低"

**第九节的原结论**：CORE-02 是误报，剔除。理由是 system prompt 要求函数式输出，extractActionType 正确支持函数式。

**三次复核发现的问题**：
查看 `ModelClient.parseResponse` 完整代码（line 183-225），发现它不仅处理 `<answer>` 包裹的函数式输出，还有**多层 JSON 兜底**：

```kotlin
// line 207-218
if (!action.startsWith("{") && !action.startsWith("do(") && !action.startsWith("finish(")) {
    val funcMatch = Regex("""(do|finish)\s*\([^)]+\)""", ...).find(content)
    if (funcMatch != null) {
        action = funcMatch.value
    } else {
        val extractedJson = extractJsonFromContent(content)  // ← JSON 兜底
        if (extractedJson.isNotEmpty()) {
            action = extractedJson  // ← 这里可能赋值为 JSON
        }
    }
}
```

并且 line 208 的检查 `action.startsWith("{")` 也明确把"action 是 JSON"当作合法路径处理。

**这说明原作者明知模型会偶尔违反 prompt 输出 JSON**——否则不需要这套 JSON 兜底。在那种场景下，`response.action` 就是 JSON 字符串（如 `{"_metadata":"do","action":"Tap","element":[100,200]}`），`extractActionType` 的正则确实匹配不到，返回 `"unknown"`。

**修正后的结论**：
- CORE-02 **不是误报**，但严重度被原报告夸大（"全部 unknown"是基于"模型总是输出 JSON"的错误假设）。
- 实际影响：当模型违反 prompt 输出 JSON 时，受影响的 step 在 DB 里 `actionType = "unknown"`，相关失败计数器 / 微信 type 提示路径会失效。
- **修订严重度**：剔除 → 低。

### 10.2 撤回：CORE-04 中"line 232-233 是死代码"的附带发现

**修正**：
