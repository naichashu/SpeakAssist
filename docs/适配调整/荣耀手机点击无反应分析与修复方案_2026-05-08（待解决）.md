# 荣耀手机点击无反应分析与修复方案（待解决）

> 创建日期：2026-05-08
> 状态：待解决（方案已制定，需在荣耀机上实测验证）

## 摘要

- **症状**：荣耀手机上点击日志显示成功，但目标 App 完全无反应；其他品牌偶发，荣耀高频。
- **直接原因**：`MyAccessibilityService.kt:128` 用 `StrokeDescription(path, 0, 100)` 派发点击，100ms 在荣耀 MagicOS 的反误触阈值（80–150ms）边界上，被系统当注入误触静默丢弃。
- **代码层面的 bug**：当前把 `dispatchGesture` 的 `onCompleted` 回调当成"点击成功"，但这个回调只代表"派发进系统"，不代表"下层 App 收到"——事件被反误触吞掉时，回调照样回 success，所以日志撒谎。
- **为什么本机不复现、换机就出**：每家 OEM 的反误触松紧不同。原生 / 三星 / OPPO 宽松，100ms 通过；华为 EMUI / 荣耀 MagicOS 最严，要 ≥120–150ms。
- **推荐方案**：**方案 A（100ms → 150ms 一刀切）**，一行改动、低风险、对所有机器都稳。B / C 是 A 不够时的预案，目前不动。
- **结论的不确定性**：根因目前是强假设（XDA 同症状报告 + 项目内 300ms swipe / 500ms longPress 不出问题、只有 100ms tap 出问题 + OEM 差异表现三条证据相互印证），最终需要在荣耀机上跑一轮验证才能 100% 实锤。

## 一、现象

在荣耀手机上执行任务循环时，多次出现以下症状：

- 日志输出 `点击手势完成` + `点击成功：坐标(x, y)`；
- 模型/`ChatViewModel` 也按"点击成功"流入下一步；
- 但**目标 App 完全无反应**，目标按钮没有被触发，UI 状态没有任何变化；
- 下一轮模型基于"点过了"的前提继续推理，导致整个任务跑偏或卡死。

非荣耀机（少量复现）也偶尔出现，但荣耀上明显高频。

## 三、根因分析

### 3.1 第一层：日志含义被高估

`onCompleted` 回调的真正语义是"**手势已经派发进系统的输入管线**"，**不**等价于"目标 App 已经收到点击事件"。

> 来源：[Android Developers — GestureDescription.StrokeDescription](https://developer.android.com/reference/android/accessibilityservice/GestureDescription.StrokeDescription)、[Mihir Patel — Android Accessibility Service: The Unexplored Goldmine](https://mhrpatel12.medium.com/android-accessibility-service-the-unexplored-goldmine-d336b0f33e30)

派发完成后还会经过：

1. 系统层 `InputDispatcher` 路由到目标窗口；
2. 厂商定制的输入预处理（防误触、防注入、防自动化等）；
3. 目标窗口的 `View.dispatchTouchEvent` → 命中 View 的 `onClick`。

任何一层将事件丢弃，对我们都是"静默失败"——回调照样回 `onCompleted`，但下层没收到。

**结论**：`点击手势完成` 这条日志是真的，但它不能用来当作"点击成功"的证据，是当前代码做了错误的等价。

### 3.2 第二层：100ms 的合成触摸被荣耀 MagicOS 当成误触/注入丢弃（最可疑根因）

证据链：

**a) XDA 论坛存在症状一致的报告**

> [some apps didn't response dispatchGesture sent by android AccessibilityService](https://xdaforums.com/t/some-apps-didnt-response-dispatchgesture-sent-by-android-accessibilityservice.4608747/)
>
> 同样代码 `StrokeDescription(path, 0, 100)`、同样回 `onCompleted`，A app 行、B app 不行；`adb shell input tap` 也不行；airtest（用了不同的注入路径或更长时长）行。

这条与本项目症状高度吻合："日志成功，下层无反应"。

**b) 项目内部对照**

| 手势 | 时长 | 在荣耀上的表现 |
|------|------|---------------|
| `tap` | 100ms | **频繁失败** |
| `swipe` | 300ms | 工作正常 |
| `longPress` | 500ms | 工作正常 |

只有最短的 100ms 单点出问题，时长越短越容易被误触过滤丢弃，与"OEM 反误触/反注入有最低时长门槛"的假设完全一致。

**c) 行为机制**

国产 OEM（华为 EMUI、荣耀 MagicOS、小米 MIUI 等）普遍在 `InputDispatcher` 之前加了一层私有的输入过滤：

- 检查事件的 source flag（`SOURCE_TOUCHSCREEN` vs `SOURCE_INJECTED`）；
- 检查 down→up 间隔（典型阈值在 80–150ms 之间）；
- 检查 down 与 up 的坐标位移（零位移可能被判误触）；
- 检查事件密度（短时间多个注入事件）。

`AccessibilityService.dispatchGesture` 在底层走的就是注入路径，所有这些过滤都会作用到它身上。100ms 单点正好踩在一个常见过滤阈值边界上。

### 3.3 排除项：Android 12+ 触摸穿透限制（不适用本场景）

> 参考：[Behavior changes: all apps — Android 12](https://developer.android.com/about/versions/12/behavior-changes-all)、[Untrusted Touch Events in Android](https://medium.com/androiddevelopers/untrusted-touch-events-2c0e0b9c374c)

Android 12+ 对 `TYPE_APPLICATION_OVERLAY + FLAG_NOT_TOUCHABLE` 的窗口确实加了"触摸穿透"限制。但这条规则是用来**阻挡用户真实触摸**穿过悬浮窗去打到下层 App，目的是防止钓鱼覆盖层欺骗用户。`dispatchGesture` 是无障碍服务从系统侧注入的事件，**不**走"用户真实触摸→穿透 overlay→落到下层"这条路径，因此不受这条规则影响。

`ExecutionCardView` 用 `FLAG_NOT_FOCUSABLE | FLAG_NOT_TOUCHABLE` 的写法本身没问题，不是当前 bug 的原因。

### 3.4 排除项：lineTo 微抖动方案

> 参考：[Android Developers — GestureDescription.StrokeDescription](https://developer.android.com/reference/android/accessibilityservice/GestureDescription.StrokeDescription)

之前考虑过给 click path 加一个 `lineTo(x+1, y)` 模拟"真实手指的亚像素抖动"，**这个方向是错的**：

- 标准 Android click path 必须只有 `moveTo`，不能有任何 `lineTo`；
- 一旦出现 `lineTo`，哪怕只挪 1px，整个手势会被分类成 drag/swipe；
- 大量 View 对 drag 与 click 走完全不同的处理逻辑，引入 `lineTo` 反而会让点击在更多 App 上失效。

这条不能做。

### 3.5 排除项：坐标系不一致

`ChatViewModel.kt:236-250` 优先取 `screenShot.width / .height` 作为 `screenWidth / screenHeight`，而 `screenShot` 来自 `AccessibilityService.takeScreenshot(DEFAULT_DISPLAY)`，是默认 display 的物理像素位图。`dispatchGesture` 接受的也是同一 display 的物理像素坐标。两者坐标系一致，不存在系统性偏移。

## 四、修复方案

按"低风险 → 高风险"分三层。

### 方案 A（推荐 / 必做、最小改动）：把 click 时长从 100ms 提到 150ms

**改动位置**：`app/src/main/java/com/example/service/MyAccessibilityService.kt:128`

```diff
- val stroke = GestureDescription.StrokeDescription(path, 0, 100)
+ val stroke = GestureDescription.StrokeDescription(path, 0, 150)
```

**理由**：

- 150ms 安全跨过常见 OEM 反误触阈值（80–150ms）；
- 仍远小于 `ViewConfiguration.getLongPressTimeout()`（默认 500ms），不会被识别成长按；真人手指点一下平均也是 80–180ms，150ms 反而更"像人"；
- 一行改动，回归风险极低。

`doubleTap` 的两次 100ms 也建议同步调整为 150ms（`MyAccessibilityService.kt:194,197`），两次按下时长保持一致。`longPress` 500ms、`swipe` 300ms 不动。

**对原本工作正常机器的影响（成本评估）**：

当前一次 tap 的真实耗时拆解：
- 50ms（chip detach 延迟）
- 100ms（手势本身）
- 50ms（chip reattach 延迟）
- **1500ms**（`TAP_SETTLE_DELAY_MS`，等 UI 稳定）
- 叠加截屏 + 模型推理 + HTTP，每步总耗时通常 4–6 秒

把手势从 100ms 改到 150ms，每步只多 50ms，相当于整体 **<1%** 的增量，肉眼不可见。本机原本没问题的机器不会有可感知的体验下降。

### 方案 B（A 不够时再叠加）：把手势前后的 50ms 延迟提到 100ms

**改动位置**：`app/src/main/java/com/example/service/MyAccessibilityService.kt:254,270`

```diff
- delay(50) // 等待芯片从窗口层完全移除后再发手势
+ delay(100) // 等待芯片从窗口层完全移除后再发手势（荣耀重负载下 50ms 偏短）
...
- delay(50) // 手势结束后稍作等待再装回芯片
+ delay(100) // 手势结束后稍作等待再装回芯片
```

**理由**：

- `WindowManager.removeView` 是异步的，重负载下 surface 真正卸载可能 >50ms；
- 如果芯片还没真正撤掉就发手势，触摸可能落到芯片上被吞（芯片在右上角 `16dp,56dp`）；
- 副作用：每步任务多 100ms，可接受。

### 方案 C（A+B 仍不够时再上）：成功语义升级——用 ACTION_CLICK 兜底

**思路**：`dispatchGesture` 派发完后，再用 `AccessibilityNodeInfo.performAction(ACTION_CLICK)` 在同一坐标对应的节点上补一发。两条路径任一成功则视为点击成功。

**改动位置**：在 `MyAccessibilityService.clickByNode` 内：

```kotlin
suspend fun clickByNode(x: Float, y: Float): Boolean {
    // ... 原有 dispatchGesture 派发逻辑

    val gestureResult = dispatchGestureAwaiting(gesture)
    Log.d(TAG, "点击手势${if (gestureResult) "派发完成" else "派发失败"}")

    // 兜底：用节点 ACTION_CLICK 再补一次（针对手势被反误触吞掉的情况）
    val nodeResult = tryClickByNode(x, y)
    Log.d(TAG, "节点点击${if (nodeResult) "成功" else "未成功（无可点击节点）"}")

    return gestureResult || nodeResult
}
```

**注意事项**（CLAUDE.md 已经说过的硬约束）：

- `findClickableNodeAt` 必须用栈式遍历 + 显式 `recycle`，**不要**用递归 DFS（会泄漏 native 内存）；
- `root` 与目标 node 之间必须用 `it !== root` 身份比较防止双重 close；
- 这条改动量较大，回归风险较高，**只在 A+B 都不够时再上**。

### 方案不做的原因（避免重复踩坑）

- ❌ 给 click path 加 `lineTo` 微抖动：会把 tap 变成 drag，反而扩大问题面（详见 §3.4）。
- ❌ 把 ExecutionCardView 改成 `TYPE_ACCESSIBILITY_OVERLAY`：当前问题不是 Android 12 触摸穿透引起（详见 §3.3），改类型属于过度修复，且 `TYPE_ACCESSIBILITY_OVERLAY` 的生命周期与 service 强绑定，会引入新一类问题。
- ❌ 全面替换为 `ACTION_CLICK`：很多 App 的关键控件并不暴露为可点击节点（自定义 View、SurfaceView、WebView 内部元素），会反而砍掉一大批原本能点的场景。
- ❌ 按 `Build.MANUFACTURER` 做设备分档（华为/荣耀 150ms、其他 100ms）：维护成本高（白名单永远写不全，下次换个机型踩同一坑还得回来加一行），而 150ms 一刀切对原本工作的机器只多 50ms（<1% 单步耗时，肉眼不可见），性价比远不如直接统一时长。

## 五、验证步骤

1. 应用方案 A（仅改 1 行 + doubleTap 同步），打 debug 包到出问题的荣耀机；
2. 在原本失败的场景上跑同一条任务，反复 10 次，统计点击成功率；
3. 同时在非荣耀机（小米/三星/Pixel）上回归冒烟，确认没回退；
4. 如果 A 把成功率从"频繁失败"拉到 ≥ 90%，结案；
5. 如果 A 仍偶发失败，叠加方案 B；
6. 如果 A+B 还不够，再考虑方案 C（提案需要单独 review）。
