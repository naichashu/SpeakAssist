# 模型卡死循环检测方案

> 创建日期：2026-05-14
> 状态：已分析、暂不实施，留作后续触发参考
> 关联文档：
> - `docs/查看型任务_finish过早与输出过短修复方案.md`（任务终止条件相关）
> - `docs/适配调整/机型点击适配方案_补强意见_2026-05-11.md`（多信号融合的早期讨论）

## 一、问题描述

模型在 `executeTaskLoop` 内陷入"每一步动作都成功、但任务毫无实质进展"的死循环，最终撞 `maxSteps=50` 才退出。两个典型 case：

| Case | 用户意图 | 实际现实 | 模型行为 | 现有兜底 |
|---|---|---|---|---|
| 7 分糖 | 点一杯 7 分糖奶茶 | 店里只有正常糖 / 少糖 / 无糖 | 反复点开糖度弹窗找"7 分糖"按钮 | maxSteps=50 |
| 单人餐 | 点一份单人餐 | 店里没有"单人餐"分类 | 一直 swipe down 翻菜单找 | maxSteps=50 |

共同点：**每一步动作都 success=true，模型没有自发判断"该放弃了"。**

## 二、现有保护为什么不够

`ChatViewModel.executeTaskLoop` 内已有的安全网：

| 保护 | 阈值 | 适用场景 | 是否覆盖本问题 |
|---|---|---|---|
| `maxSteps` | 50 | 总步数兜底 | 能拦住但要等 50 步，用户体验差 |
| `maxRetries` | 4 | 连续动作**失败**重试 | 不覆盖——动作签名相同但每次 success=true 不计数 |
| `stepActionFailures` | per action type | 同种 action 的失败次数 | 同上，每次 success 后清零 |
| system prompt 第 10 条 | 自然语言 | 让模型自己判断"已到边界" | 模型能力上限，AutoGLM 当前版本不可靠 |

漏洞核心：**所有现有计数器都是失败计数器，没有"成功但无进展"计数器。**

## 三、方案：双信号融合

学术 SOTA 做法（Mobile-Agent 论文）。两个独立信号同时满足才触发：

### 信号 1：动作签名重复

把每个执行成功的 action 归一化成签名字符串：

```
tap     → "tap_${x/50}_${y/50}"                # 坐标按 50 像素粒度归一
swipe   → "swipe_${主方向}"                    # up/down/left/right
type    → "type_${text}"                       # 完整文本
其他    → "${actionType}"
```

维护 `lastSignature` 和 `sameSignatureStreak`，签名连续相同则计数+1。

### 信号 2：截图前后相似度

每步动作执行**后**重新抓一张截图，与执行**前**那张比对：

- 算法：[pHash](https://en.wikipedia.org/wiki/Perceptual_hashing)（感知哈希）
- 步骤：缩到 32×32 灰度 → DCT 取低频分量 → 计算 64-bit hash
- 相似度：Hamming distance ≤ 5（约 92% 相似）视为"屏幕没变化"
- 截图来源：`captureForegroundContext` 已经在每步抓，缓存上一步的结果即可，无额外 IPC

### 触发条件

```
sameSignatureStreak >= 3  AND  screenshot_phash_distance <= 5
```

两者**同时**满足时调用 `finishTask(sessionId, false, "连续 3 次执行 '$signature' 屏幕无变化，疑似卡死")`。

### 为什么是"同时"

| 场景 | 签名重复？ | 截图变化？ | 触发？ |
|---|---|---|---|
| 7 分糖反复点弹窗 | ✓ | ✗（同一弹窗） | ✓ |
| 单人餐 swipe 到底 | ✓ | ✗（已到边界） | ✓ |
| 长菜单正常浏览 | ✓ | ✓（每屏新内容） | ✗ |
| 删除多条通知 | ✓ | ✓（通知少一条） | ✗ |
| 抢购按钮轮询 | ✓ | ✓（状态在更新） | ✗ |

单信号会误伤后三种合法场景，双信号融合显著降低误伤率。

## 四、实现位置与代码量

- 文件：`app/src/main/java/com/example/ui/viewmodel/ChatViewModel.kt`
- 位置：`executeTaskLoop` 内、`result.success=true` 分支（约 344 行附近）
- 新增工具类：`com/example/util/PerceptualHash.kt`（~30 行）
- 总代码量：~60 行
- 无新依赖（pHash 用 Android Bitmap 自带 API + 简单 DCT 实现）

代码草图：

```kotlin
// while 循环外初始化
var sameSignatureStreak = 0
var lastSignature: String = ""
var lastScreenshotHash: Long? = null

// success 分支
val signature = buildActionSignature(actionType, result.actionDetail)
val newHash = screenShot?.let { PerceptualHash.compute(it) }
val screenSame = lastScreenshotHash != null && newHash != null &&
                 PerceptualHash.hammingDistance(lastScreenshotHash!!, newHash) <= 5

if (signature == lastSignature && screenSame) {
    sameSignatureStreak++
    if (sameSignatureStreak >= 3) {
        return finishTask(sessionId, false,
            "连续 ${sameSignatureStreak} 次执行 '$signature' 屏幕无变化，疑似卡死，建议人工接管")
    }
} else {
    sameSignatureStreak = if (signature == lastSignature) sameSignatureStreak + 1 else 1
}
lastSignature = signature
lastScreenshotHash = newHash
```

## 五、为什么 2026-05-14 决定**暂不实施**

### 5.1 阈值无数据支撑

- "Hamming distance ≤ 5" 是经验初值，没有跑过任何真实任务校准
- "连续 3 次" 同理
- 校准只能上线后收集真实任务数据，意味着首发版本必然不准

### 5.2 pHash 在 mobile UI 上的行为未验证

- pHash 设计目标是通用图像（抗 JPEG 压缩、轻微旋转），不是 UI
- UI 的"真变化"可能极小（列表多一项、按钮变色）→ pHash 可能判"相同"，误触发
- UI 的"假变化"也常见（时间更新、通知红点闪烁、loading 旋转图标）→ pHash 可能判"不同"，抓不到真卡死

### 5.3 没有 ablation 数据

- 不知道真实任务里这种"成功但卡死"占多大比例
- 不知道改完后挽救率 vs 误伤率
- 没有 case 库去回归验证

### 5.4 收益 vs 成本不对称

- 即使方案正确，挽救体验也只是 "从 50 步 finish 改成 3 步 finish"——用户都得重新发起任务
- 模型 reasoning 能力问题没解决，只是早一点 finish
- 客户端引入一块需要长期维护的代码

### 5.5 模型在迭代

- AutoGLM 在持续升级，"卡死场景"的本质是模型推理能力问题
- 等下一代模型可能直接消失
- 提前加客户端逻辑，模型升级后反而成包袱

## 六、什么时候回来做（触发条件）

满足以下**任一**条件再实施：

1. **真实任务数据显示卡死高频** —— 例如统计显示 ≥ 5% 的任务撞到 `maxSteps=50` 才结束，且复盘后归因为"模型卡死"而非"任务本身复杂"。
2. **用户主动反馈痛点** —— 多个独立用户报告"任务一直停不下来"。
3. **模型升级后仍未解决** —— AutoGLM 至少升级一个大版本后，同样场景仍然卡死。
4. **需要更稳的体验保障** —— 项目从毕业设计走向生产环境，对用户体验稳定性要求提高。

## 七、实施时的检查清单

如果未来真的要做，按下面顺序：

1. **先加日志，不动行为** —— 计算双信号但只 `Log.w`，跑两周收数据，看真实命中分布
2. **校准阈值** —— 根据日志统计调整 `≤5 / 3 次` 的具体值
3. **加 debug 开关** —— 默认关闭，通过 `BuildConfig.DEBUG` 或硬编码 const 控制
4. **小流量上线** —— 自己用一阵子观察误伤
5. **全量打开** —— 确认稳定后改默认值

## 九、附：为什么不选 prompt 路线

学术界主流是双信号 + reflection prompt（"你好像在重复，试试别的"）。本方案**不走 prompt 路线**，原因：

- system prompt 已经很长（12 条规则），每加一条都增加 token 成本和模型理解负担
- prompt 改动是软约束，模型可以选择不听
- 客户端硬约束行为更可预测、更可调试
- 真要做 reflection，效果不如直接 finish 让用户接管——毕竟点单场景容错率低
