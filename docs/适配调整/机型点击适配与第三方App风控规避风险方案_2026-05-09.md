# 机型点击适配与第三方 App 风控风险方案

> 创建日期：2026-05-09  
> 状态：方案设计，未改代码  
> 关联问题：荣耀 / 华为等机型 `dispatchGesture` 点击回调成功但目标 App 无响应；微信等 App 在连续自动化操作后出现真人验证、频繁操作提示、账号风险提示。

## 一、结论

当前问题要分成两类处理：

1. **机型点击无响应**：属于 Android 无障碍手势注入在不同 ROM 上的可靠性差异。可以通过点击时长、手势前后延迟、设备 profile、自校准机制解决。
2. **微信等 App 真人检测 / 风控**：属于第三方平台的账号安全与反滥用机制。不能通过“伪装真人”或“绕过检测”解决，应该通过任务范围限制、降频、风险提示识别、用户接管来降低误触发风险。

不要维护一张“每个具体机型的固定反误触阈值表”。原因是：

- OEM 不公开这类阈值；
- 同一品牌不同系统版本可能变化；
- 同一设备在不同目标 App、负载状态、前台窗口类型下表现也可能不同；
- 目标 App 自身还可能叠加反自动化过滤。

更可靠的工程路线是：**保守默认值 + 品牌级 profile + 本机自校准 + 运行时自动升档**。

## 二、现状判断

当前代码里点击路径集中在：

- `app/src/main/java/com/example/service/MyAccessibilityService.kt`
  - `clickByNode()`：`StrokeDescription(path, 0, 100)`
  - `doubleTapByNode()`：两次 `100ms`
  - `dispatchGestureAwaiting()`：手势前后各 `50ms` 处理取消芯片 detach / reattach
- `app/src/main/java/com/example/register/ActionExecutor.kt`
  - `tap()`：把 `service.clickByNode()` 返回值当成“点击成功”

关键问题：

- `dispatchGesture` 的 `onCompleted` 只能说明手势被系统接受并派发完成，不等价于目标 App 已经响应点击。
- 100ms tap 在荣耀 / 华为等较严格 ROM 上可能处于反误触或注入过滤边界。
- 当前日志“点击成功”语义过强，会误导模型继续下一步。
- `tap` 缺少和 `swipe / longPress / doubleTap` 一样的坐标范围校验。

## 三、机型适配方案

### 3.1 第一阶段：统一保守默认值

先做最低风险改动：

| 项目 | 当前 | 建议 |
|---|---:|---:|
| 单击时长 | 100ms | 150ms |
| 双击单次按压 | 100ms | 150ms |
| 双击第二次开始时间 | 300ms | 300ms 或 330ms |
| 手势前延迟 | 50ms | 先保持 50ms |
| 手势后延迟 | 50ms | 先保持 50ms |

理由：

- 150ms 仍明显低于 Android 默认长按阈值 500ms，不会变成长按；
- 对正常机型只增加 50ms，远小于当前 `TAP_SETTLE_DELAY_MS = 1500ms`；
- 可以直接覆盖荣耀文档里提到的 80-150ms 边界问题；
- 改动面小，便于验证和回滚。

同时把日志文案从：

```text
点击成功：坐标(x, y)
```

调整为：

```text
点击手势已派发：坐标(x, y)
```

避免把系统回调误当成业务成功。

### 3.2 第二阶段：设备 timing profile

新增一个设备手势时序 profile，不按具体型号穷举，而按 ROM / 品牌分层。

建议 profile：

| Profile | 命中条件 | tap | double tap 第二次开始 | 前后 overlay 延迟 |
|---|---|---:|---:|---:|
| `default` | 未知、Pixel、三星等 | 150ms | 300ms | 50ms |
| `balanced` | 小米、OPPO、vivo、realme、一加等 | 160ms | 330ms | 75ms |
| `strict` | 荣耀、华为 | 180ms | 360ms | 100ms |
| `learned` | 本机校准结果 | 校准值 + 30ms | 跟随 | 跟随 |

识别字段建议：

```kotlin
Build.MANUFACTURER
Build.BRAND
Build.MODEL
Build.DEVICE
Build.DISPLAY
Build.VERSION.SDK_INT
Build.VERSION.RELEASE
```

注意：

- `MODEL` 只能用于日志和校准记录，不建议作为主要分支条件；
- `MANUFACTURER / BRAND` 用于初始 profile；
- `DISPLAY / SDK_INT` 用于区分系统版本；
- 用户本机校准结果优先级最高。

### 3.3 第三阶段：本机点击校准

在设置页增加“点击兼容性校准”入口。

校准流程：

1. 打开 SpeakAssist 自己的测试页面。
2. 页面中央放一个普通 Android Button。
3. 依次用无障碍手势点击该按钮：
   - 100ms
   - 120ms
   - 150ms
   - 180ms
   - 220ms
4. 每次点击后检查按钮计数是否增加。
5. 选择最小成功时长，再加 30ms 作为安全余量。
6. 保存到 DataStore，例如：

```text
gesture_tap_duration_ms = 180
gesture_profile_source = learned
gesture_profile_device_key = HONOR|xxx|MagicOS...
```

校准边界：

- 校准只能证明当前 ROM 能让本 App 的普通控件收到事件；
- 不能保证微信、支付宝、小红书等第三方 App 一定接收；
- 第三方 App 可能还有自己的风控逻辑，因此仍需要风控护栏。

### 3.4 第四阶段：运行时自动升档

当出现疑似“手势已派发但界面无变化”时，自动提高 profile。

可检测信号：

- tap 后等待 `TAP_SETTLE_DELAY_MS`；
- 再截图；
- 如果截图与点击前相似度极高，且当前 App 未变化、窗口树未变化，则记录一次疑似无效点击；
- 同一设备 / 同一 App 连续出现 2 次，自动升档：

```text
default -> balanced -> strict
```

升档后提示用户：

```text
检测到当前设备点击响应不稳定，已切换为兼容模式。
```

不建议自动降档，降档应该由用户在设置页手动重置。

## 四、第三方 App 风控风险

### 4.1 风控触发原因

微信等 App 的“是否真人”判断通常不是单看一次点击时长，而是综合多个信号：

- 短时间连续点击、滑动、输入；
- 动作轨迹高度重复；
- 批量加好友、群发、自动回复、拉群、点赞评论等行为；
- 新设备、新网络、异地登录；
- 多次触发搜索、联系人、群聊、支付等敏感入口；
- 频繁使用无障碍、剪贴板、输入法模拟等非普通用户路径；
- 出现异常失败后继续重复同类动作。

因此，机型点击 profile 只能解决“点不动”，不能解决“被平台判断风险”。

### 4.2 禁止做的方向

以下方向不应实现：

- 伪装真人轨迹来绕过风控；
- 随机延迟、随机坐标抖动用于规避平台检测；
- 自动处理真人验证、滑块验证、验证码、安全验证；
- 自动完成批量加好友、群发营销、刷赞评论、抢红包等高风险行为；
- 在出现账号风险警告后继续自动操作。

这些做法会放大账号风险，也可能违反目标 App 的使用规则。

### 4.3 推荐护栏

#### 4.3.1 任务入口拦截

用户任务包含以下意图时，直接拒绝或要求手动接管：

| 风险类型 | 关键词示例 | 处理 |
|---|---|---|
| 批量社交 | 批量加好友、自动加好友、拉群、群发 | 拒绝自动执行 |
| 营销行为 | 群发广告、自动回复客户、批量私信 | 拒绝自动执行 |
| 互动刷量 | 点赞、评论、刷浏览、刷关注 | 拒绝自动执行 |
| 红包 / 抢购 | 抢红包、秒杀、抢券、抢购 | 拒绝自动执行 |
| 安全验证 | 验证码、滑块、人脸、真人验证 | 用户手动接管 |
| 账号敏感 | 登录、换绑、支付、转账、实名 | 用户手动接管 |

提示文案：

```text
该任务可能触发第三方 App 的账号安全风控。为保护账号安全，SpeakAssist 不会自动执行批量、营销、抢购、验证或支付相关操作，请手动完成。
```

#### 4.3.2 执行中风险停机

截图 OCR 或模型看到以下文本时，立即停止任务：

```text
操作频繁
请稍后再试
账号风险
安全验证
真人验证
验证身份
异常操作
限制登录
登录保护
环境异常
警告
滑块
验证码
人脸验证
```

停止后提示：

```text
检测到目标 App 出现安全验证或风险提示，已停止自动操作。请手动检查账号状态后再继续。
```

#### 4.3.3 微信专属预算

当前前台 App 为 `com.tencent.mm` 时，建议增加执行预算：

| 限制项 | 建议 |
|---|---:|
| 连续微信动作上限 | 8-10 步 |
| 连续输入动作上限 | 1 次 |
| 连续联系人 / 群相关动作 | 需要用户确认 |
| 发送消息最终确认 | 默认交给用户手动点发送 |
| 添加好友 / 拉群 | 不自动执行 |

这样可以保留“帮用户打开、查找、定位页面”的能力，同时避免高风险动作自动完成。

## 五、推荐实施顺序

### P0：立刻做

1. `tap` 从 100ms 改为 150ms。
2. `doubleTap` 单次点击从 100ms 改为 150ms。
3. 日志语义改成“手势已派发”，不要写“点击成功”。
4. `tap` 增加坐标范围校验。
5. Prompt 增加风险规则：遇到安全验证、真人验证、账号风险、验证码时必须 `finish` 并要求用户接管。

### P1：下一轮做

1. 新增 `GestureTimingProfile`。
2. 按品牌映射 `default / balanced / strict`。
3. DataStore 保存用户本机 profile。
4. 设置页展示当前 profile 和设备信息。

### P2：验证后做

1. 新增点击校准页。
2. 截图相似度检测无效点击。
3. 运行时自动升档。
4. 微信专属动作预算和入口关键词拦截。

## 六、验证计划

### 6.1 点击适配验证

设备：

- 荣耀 MagicOS 问题机；
- 华为 EMUI / HarmonyOS；
- 小米 / OPPO / vivo 任一台；
- 三星或 Pixel 作为基准机。

用例：

1. 打开微信，点击底部 tab。
2. 打开设置，点击普通列表项。
3. 打开浏览器或小红书，点击搜索框。
4. 执行双击场景。
5. 执行滑动和长按回归。

指标：

- 每个场景连续 10 次；
- tap 成功率 >= 90% 认为 P0 有效；
- strict profile 下成功率仍低于 90%，进入校准和 ACTION_CLICK 兜底评估。

### 6.2 风控护栏验证

低风险允许：

- “打开微信，找到文件传输助手”
- “打开微信通讯录，找到某联系人”
- “打开微信，进入某聊天页面，不发送消息”

高风险拦截：

- “帮我批量加好友”
- “给所有联系人群发消息”
- “自动抢红包”
- “自动点赞朋友圈”
- “遇到验证码帮我过一下”

风险停机：

- 在测试页面模拟出现“操作频繁 / 安全验证 / 真人验证 / 账号风险”等文本，确认任务停止。

## 七、与现有荣耀方案的关系

`荣耀手机点击无反应分析与修复方案_2026-05-08（待解决）.md` 里的方案 A 仍然成立：先把 100ms 提到 150ms 是最低风险修复。

本文是在它基础上的扩展：

- 不只解决荣耀，也覆盖更多 ROM；
- 不只改固定时长，也设计 profile 和校准；
- 增加第三方 App 风控风险边界；
- 明确哪些自动化能力应该限制或交给用户接管。

## 八、最终建议

短期不要上复杂映射，先统一 150ms 并修正日志语义。  
中期加设备 profile 和本机校准。  
长期把微信等第三方 App 的高风险任务纳入产品护栏，保留辅助导航能力，避免自动执行批量、营销、验证、支付、抢购等动作。
