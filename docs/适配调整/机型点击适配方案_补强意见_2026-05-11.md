## 零、整体评价

主方案的判断是对的：

- 分清「机型点不动（手势注入可靠性）」与「被平台风控（账号安全）」是两类完全不同的问题；
- 不维护机型白名单这种长期会失控的方案；
- 用「保守默认值 + 品牌 profile + 本机校准 + 运行时升档」的组合而不是一刀切。

下面是几处会影响落地效果或长期维护性的调整建议，按重要性排序。

## 一、对 05-09 方案的调整

### 1.1 P0 时长从 150ms 调到 180ms 起步、按品牌分三档

主方案 P0 是把 100ms 改 150ms。两点不足：

- 05-08 文档引用的「OEM 反误触阈值 80–150ms」**没有官方依据**，仅来自 [XDA 同症状报告](https://xdaforums.com/t/some-apps-didnt-response-dispatchgesture-sent-by-android-accessibilityservice.4608747/)和项目内对照实验，是经验性推断；
- 150ms 接近真人 tap 中位数（133ms ± 83ms，[Asakawa et al. 2017](https://www.sciencedirect.com/science/article/abs/pii/S1050641116301225)，n=13 触屏平板），"像真人"但没余量。

**有官方依据的硬数据：**

| 项 | 值 | 来源 |
|---|---:|---|
| Android `ViewConfiguration.getTapTimeout()` 默认值 | 115ms | [Android framework 源码](https://android.googlesource.com/platform/frameworks/base/+/master/core/java/android/view/ViewConfiguration.java) |
| Android `getLongPressTimeout()` 默认值 | 500ms | 同上 |
| Android `getDoubleTapTimeout()` 默认值 | 300ms | 同上 |
| 真人 tap 实测均值 | 133ms ± 83ms | [Asakawa et al. 2017](https://www.sciencedirect.com/science/article/abs/pii/S1050641116301225) |

**仍是推断、查不到官方依据的：**

- 国产 OEM 对 dispatchGesture 注入事件的具体过滤阈值（仅有 XDA 同症状报告 + 项目内 100ms tap 失败 / 300ms swipe 正常的对照）；
- 华为需要比小米更长这个具体分档（基于"华为机已知失败"这一故障数据点，非官方测试）。

**建议（按真人 tap 分布选取）：**

| Profile | 命中条件 | tap 时长 | 对应真人 tap 百分位 |
|---|---|---:|---:|
| `default` | 未知品牌、Pixel、三星 | 180ms | p70 |
| `balanced` | 小米/OPPO/vivo/realme/一加 | 200ms | p78 |
| `strict` | 华为/荣耀（已知故障点） | 220ms | p85 |

- 三档都在真人 tap 自然分布内（p70-p85）→ 即使第三方 App 有"非真人"检测也不显异常；
- 三档都**远低于**长按 500ms → 不会被识别为长按；
- 三档都**长于**目前已知会失败的 100ms → 跨过了观察到的失败区间（XDA 同症状报告 + 项目内 100ms tap 失败 / 300ms swipe 正常的对照）；
- doubleTap 第二次开始 350/360/380ms → 与第一次 up 的间隔分别 170/160/160ms，均在 doubleTapTimeout 300ms 内，会被识别为双击。

注：Android 标准 `getTapTimeout()` 默认 115ms，是"等多久判断是 tap 还是 scroll"的窗口，**不是**事件被丢弃的阈值。100ms 在 Android 标准逻辑下仍会被判为 tap——失败发生在 OEM 私有的注入事件过滤层，其具体规则未公开。

**对正常机器的体验影响**：单步多 80-120ms，占总耗时（4-6s）<3%，肉眼不可见。

**具体值的局限**：分三档以及 180/200/220 的具体值**没有"测出来"的依据**——只是基于真人 tap 分布 + 已知故障点的合理初值。真正最优值需在真实设备上跑成功率统计调整。

改动位置：

- `MyAccessibilityService.kt:clickByNode`：100ms → `profile.tapDurationMs`
- `MyAccessibilityService.kt:doubleTapByNode`：两次 100ms + 间隔 300ms → profile 全套
- 新增 `service/GestureTimingProfile.kt`：3 档静态配置 + Build.BRAND/MANUFACTURER 选档

### 1.2 自校准的可信度问题

3.3 节的校准设计是「在 SpeakAssist 自己的页面放一个 Button，连续点不同时长找最小成功值」。这个方案乐观了。原因：

- 自身 App 进程内的 view 不会经过任何反自动化过滤，自校准成功不能外推到第三方 App；
- 部分国产 App（支付、银行、社交大厂）在 `View.dispatchTouchEvent` 内还有自己的 `SOURCE_INJECTED` 二次校验；

建议调整：

- **校准对象**改成系统设置里的一个 toggle（系统设置不做反自动化拦截，但跨过了「自身 App 进程」边界，更接近真实场景）；
- **校准结果**只作为 profile **初值**，真正可信的是线上观察到的失败率（见 1.3、2.3）；
- 不把校准通过当作「已免责，不需要兜底」的状态。

### 1.3 失效点击检测改成多信号融合

3.4 节用截屏相似度作为唯一信号判断「点击是否生效」。会有这几类误报：

- 点击已选中 tab，UI 本来就不变；
- 点击 toggle 但视觉差异极小；
- 点击触发后台请求，画面要等 loading 才有变化；

建议融合 4 个信号，任一变化即视为点击生效：

| 信号 | 来源 | 适合场景 |
|---|---|---|
| `windowId` 变化 | `AccessibilityWindowInfo.getId()` | 跳页类点击最显著 |
| focused node 变化 | `findFocus()` 节点 hash | 表单、输入框点击 |
| contentChange 事件 | `onAccessibilityEvent.TYPE_WINDOW_CONTENT_CHANGED` | 局部刷新类点击 |
| 截屏像素差比例 | 简单的逐像素差异比例（无需引入图像库） | 兜底视觉变化 |

四个信号全无变化才记一次「疑似无效点击」。

更稳的做法是**让模型自评**：在 system prompt 里要求模型每一轮 thinking 先输出 `prev_step_effective: true / false / unsure`。模型看了新截图后比对就知道，自然规避以上误报，也不需要在客户端写复杂的判定逻辑。

### 1.4 detach / reattach 改成事件驱动

主方案 §4.3（B 方案）建议 50ms → 100ms。问题：`removeView` 是异步的，重负载机上 100ms 也可能不够，固定 delay 是猜测。

建议改事件驱动：

- detach：`View.OnAttachStateChangeListener.onViewDetachedFromWindow` 回调里 resume；
- reattach：监听 `onViewAttachedToWindow`；
- 200ms 超时兜底（防回调丢失时卡死）；

这样保证芯片真的撤掉之后再发手势，不依赖固定时长猜测，对高负载机更鲁棒。

### 1.5 风险词改成"模型主导 + 关键词兜底"

主方案 §4.3.2 是硬匹配字符串列表。会漏的 case：

- 文案变体（"操作过于频繁" vs "操作太频繁"）；
- 图片化的安全验证页（OCR 才能识别文字）；
- toast 消失太快没截到；

建议两层：

- **主路径**：把检测规则写进 `strings.xml` 的 `system_prompt_template`，让模型在每一轮 thinking 里先判断"当前页面是否出现账号风险 / 真人验证 / 操作频繁 / 验证码 / 滑块 / 人脸验证"，命中即输出 `finish` 并要求用户接管；
- **兜底**：当前的字符串列表，仅当模型漏判时触发。

模型作为主检测器的好处是泛化能力强，看到没见过的变体也能识别。

### 1.6 微信预算加时间窗口维度

主方案 §4.3.3 的"连续 8–10 步上限"如果不带时间限制意义不大。30 秒内 8 步 vs 5 分钟内 8 步，触发风控概率差几个量级——微信侧的频次判断几乎都是滑动窗口。

建议改为二维约束（任一命中即强制 wait 或交还用户）：

| 时间窗 | 上限步数 | 触发动作 |
|---|---:|---|
| 60 秒 | 6 步 | 强制 wait 60s 或交还用户 |
| 5 分钟 | 15 步 | 强制 wait 60s 或交还用户 |
| 10 分钟 | 25 步 | 提示用户保存进度 |

## 二、05-09 方案缺失的内容

### 2.1 WebView / H5 场景的边界

微信小程序、淘宝商品页、小红书帖子、抖音评论区——这些页面里的可点击元素，AccessibilityService 拿到的节点树几乎为空。意味着：

- 主方案 §C（`ACTION_CLICK` 兜底）在这些场景**完全失效**；
- WebView 必须依赖手势注入路径，profile 升档不可回避；
- 升档到 strict 仍失败的，应直接 finish 并提示"当前页面为 H5 容器，建议手动操作"。

建议在 05-09 §3.4 排除项里补一条「WebView 内不可用节点兜底」，避免后续误以为"加了 C 方案就万事大吉"。

### 2.2 用户接管后的回流路径

任务被风控护栏中断 → 用户手动完成验证 / 敏感操作 → **之后呢？**

主方案止步于 `finish`，用户得重新发起完整任务并自己跳到那一步——体验差。建议设计一个「我已处理，请继续」的悬浮入口：

- 中断时保留 `taskSessionId` 和最近一次 thinking；
- 入口点击后从当前截图重新规划后续步骤；
- 重新进入后第一步必须是 `wait` 一轮，等用户操作的结果落地（避免立刻读到中间态截图）；
- 设个 30 分钟超时，超时后清掉缓存。

### 2.3 本地遥测与设置页透明

主方案讨论了 profile 自动升档、风险护栏，但没有提怎么收集这些动作的统计。后续如果想知道"strict 阈值是不是合理""哪些 App 风险词命中最多"，没数据可看。

建议加一份本地遥测：

- DataStore 记录：当前 profile、各 profile 升档次数、各类失败原因计数、最近一次失败的 App 包名和时间；
- 设置页透明展示：「本机 profile：strict（自动升档自 default）」「累计触发风险停机 X 次」；
- 不上报远程，纯本地数据，用户可手动重置；

这是后续阈值调优的唯一一手数据来源，也能让用户理解为什么自己的机器更慢一点。

## 四、修订后的落地顺序

### P0（立即做，一次发版）

1. tap 100 → **180ms**（不是 150ms）；
2. doubleTap 两次 100 → 180ms、第二次开始 300 → 350ms；
3. 日志"点击成功"改"点击手势已派发"；
4. tap 补坐标越界校验（与 swipe / longPress 对齐）；
5. system prompt 加风险规则：见安全验证 / 真人验证 / 账号风险 / 操作频繁 / 验证码 / 滑块 / 人脸验证立即 finish 并提示用户接管；
6. `Build.BRAND` 3 档 profile（10 行 when，default 180 / balanced 200 / strict 220）；
7. 失效点击日志（不升档、只 `Log.w` + 本地计数，给后续看数据）。

### P1（一周观察数据后做）

1. 失效点击的多信号融合（windowId / focused / contentChange / 像素差）；
2. detach / reattach 改事件驱动（OnAttachStateChangeListener）；
3. 设置页透明展示本机 profile 与计数；
4. 微信时间窗口预算；
5. 风险词改成"模型主导 + 关键词兜底"。

### P2（验证后做）

1. 系统设置 toggle 校准入口；
2. 运行时自动升档（基于 P1 的多信号检测）；
3. `ACTION_CLICK` 兜底（先确认 P0 + P1 不够再上）；
4. 用户接管后的「继续任务」入口。

## 五、对当前华为机问题的预期

把 P0 全部做完，**理论预期**（无设备实测数据支撑）：

- 普通 App 点击（设置、列表项、底部 tab、搜索框、对话气泡）成功率应有显著提升；
- 论据：100ms 在真人 tap 分布的 p33（偏短一端），220ms 抬到 p85（自然范围内更稳的位置）。OEM 反误触/反注入过滤的具体阈值未公开，但已知 100ms 失败、300ms swipe 正常，把单击拉到 220ms 跨过了原来的失败区间；
- **真正成功率多少只能跑数据**——不要相信我现在给的任何百分比。

仍会失败的场景：

- WebView 内点击（小程序、商品页、H5）—— 节点树几乎为空，方案 C `ACTION_CLICK` 也救不回；
- 个别风控严的 App 二次过滤（支付、银行）—— 在自己进程内校验 `SOURCE_INJECTED`，时长再长也过不去；
- 自定义 View / SurfaceView。

如果 P0 改完仍频繁失败，回到 §1.3 的失效点击日志看具体在哪一步、哪个 App 包名失败：

- 同一个 App 反复失败 → 该 App 自带过滤，需要在产品层提示用户手动操作；
- 多个 App 都失败 → 时长仍不够，把 strict 上调到 250-280ms 再试；
- 跨 App 没有规律 → 可能不是时长问题，去看 P2 的 `ACTION_CLICK` 兜底。
