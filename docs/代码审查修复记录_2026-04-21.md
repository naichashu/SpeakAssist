## 第一批：关键 Bug 修复

### 1. 截图崩溃隐患 + 无超时挂死
**文件**：`app/src/main/java/com/example/service/MyAccessibilityService.kt`

**问题**：
- `onSuccess` 回调里顺序是 `wrap(buffer) → close(buffer) → copy(bitmap)`。`close` 之后再 `copy` 语义未定义，偶发崩溃。
- `getScreenshotSuspend` 用 `suspendCancellableCoroutine` 等回调，没有超时；系统偶发卡住 `takeScreenshot`，整个任务循环永久挂起。

**修复**：
- 改为 `wrap → copy → finally close`，close 放到 `finally` 保证即使 copy 抛异常也能释放。
- `suspendCancellableCoroutine` 外包一层 `withTimeoutOrNull(5000)`，超时返回 null，上层触发"无法获取截图"的错误分支。

### 2. 唤醒词误触发 + CancellationException 被当错误
**文件**：`app/src/main/java/com/example/speech/WakeWordListeningManager.kt`

**问题**：
- `WAKE_WORDS` 列表里混了 `"小度小度"`（跟百度小度音箱冲突）、`"小猪"`（过短极易误触）、`"小路上想不想"`（无意义噪声）等条目，明显是 debug 留下的。
- 外层 `catch (e: Exception)` 捕获到 `CancellationException`（协程正常取消信号）时，把它作为"监听异常"弹给用户，用户在关闭唤醒时看到错误提示。

**修复**：
- 唤醒词列表精简为 6 个近音变体：`小噜小噜 / 小陆小陆 / 小鹿小鹿 / 小路小路 / 晓露晓露 / 小露小露`。
- 在 `catch (e: Exception)` 之前加 `catch (e: CancellationException) { throw e }`，让协程取消信号正常传递。

### 3. 聊天消息显示两次
**文件**：`app/src/main/java/com/example/speakassist/MainActivity.kt`

**问题**：
从主界面发送的每条指令，用户消息和执行状态都会被显示两遍：
1. `sendMessage` 先本地 `chatMessages.add(userMessage)`。
2. `executeTask` 里 `addSystemMessage("正在执行：$command")`。
3. `ChatViewModel.executionState` 变更后，`observeExecutionState` 又 `addUserMessage` + `addSystemMessage`。

此外还发现：
- `addTestMessages` 永久塞两条出厂消息 `"你好！我是 SpeakAssist..."`，每次 `onCreate` 都加，明显 debug 残留。
- `checkPermissions` 内联了一份 `isAccessibilityServiceEnabled` 的逻辑，和第 535-549 行的同名方法完全重复。
- Activity 销毁重建时，`observeExecutionState` 会收到 StateFlow 的最新值，完成消息被重放。

**修复**：
- 让 `observeExecutionState` 成为消息显示的唯一入口。`sendMessage` 只清输入框并触发 `executeTask`；`executeTask` 只 launch 任务，不再自己添加消息。
- `observeExecutionState` 里用 `lastCompletionKey` 记录最近一次完成状态，key 未变则跳过，防止 Activity 重建后重放。
- 删掉 `addTestMessages` 及其调用。
- `checkPermissions` 的内联逻辑替换为调用 `isAccessibilityServiceEnabled()`。

### 4. AppRegister 映射不清理
**文件**：`app/src/main/java/com/example/register/AppRegister.kt`

**问题**：
`initialize(context)` 每次任务开始都被调一次，但 `appToPackageMap` 和 `packageToAppMap` 只追加不清理：
- 卸载的应用条目永远残留，`getPackageName` 可能返回失效的包名。
- 同一应用 `appName` 和 `appName.lowercase()` 都存一份，重复。

**修复**：
- `initialize` 开头先 `appToPackageMap.clear()` 和 `packageToAppMap.clear()`。
- 删掉每个应用都打一行的 `Log.d` 噪声。

### 5. 脆弱的 finish 判断
**文件**：`app/src/main/java/com/example/ui/viewmodel/ChatViewModel.kt`

**问题**：
`isFinishAction` 已经基于 `_metadata` 识别过 finish 动作，紧接着又有一段：

```kotlin
val isFinished = result.message != null &&
    (result.message.contains("完成") || result.message.contains("finish"))
if (isFinished) {
    return finishTask(sessionId, true, ...)
}
```

任何包含"完成"或"finish"字样的错误信息（如"完成度不够"、"未找到 finish 入口"）都会被误判为任务成功，整个循环提前退出。

**修复**：删掉这段判断；finish 只信任 `_metadata == "finish"` 这条路径。

## 第二批：冗余与主线程阻塞清理

### 6. TextInputExecutor 主线程阻塞
**文件**：`app/src/main/java/com/example/input/TextInputExecutor.kt`

**问题**：
`inputText` 每次都用 `runBlocking { SettingsPrefs.textInputMode(...).first() }` 读 DataStore。`ActionExecutor.type` 从无障碍服务主线程调用，`runBlocking` 把主线程卡住等磁盘 IO。

**修复**：
- `inputText` 改为 `suspend fun`。
- `ActionExecutor.type` 同步改为 `suspend fun`（外层 `processActionObject` 本就是 suspend，调用栈贯通）。
- 删掉 `currentMode()` helper 和 `runBlocking` 依赖。

---

### 7. ActionExecutor 重复实例化 + 冗余 delay
**文件**：`app/src/main/java/com/example/register/ActionExecutor.kt`

**问题**：
- `type` 每次都 `TextInputExecutor(service, service).inputText(text)`，重新构造 `AccessibilityTextInput` 和 `ImeTextInput`。
- `launchApp` 里在 `service.startActivity(intent)` 之后立即 `delay(1000)`，然后返回时 `waitMs = LAUNCH_SETTLE_DELAY_MS (2500)`，`ChatViewModel` 根据 waitMs 又等一次。总共多等了 1 秒。

**修复**：
- `ActionExecutor` 持有 `private val textInputExecutor = TextInputExecutor(service, service)` 字段，全生命周期复用。
- 删掉 `launchApp` 内部的 `delay(1000)`，统一由上层 `waitMs` 控制启动缓冲时间。

---

### 8. Activity 权限检查双重执行
**文件**：`app/src/main/java/com/example/speakassist/MainActivity.kt`、`app/src/main/java/com/example/speakassist/SettingsActivity.kt`

**问题**：
两个 Activity 都在 `onResume` 和 `onWindowFocusChanged(hasFocus=true)` 里各调一次权限刷新，`onWindowFocusChanged` 紧随 `onResume`，等于同一件事做两次。

**修复**：
- MainActivity 的 `onResume` 不再调 `checkPermissions`，统一由 `onWindowFocusChanged` 触发。
- SettingsActivity 去掉 `onResume { onImeStateChanged() }`，同样交给 `onWindowFocusChanged`。

---

### 9. ExecutionCardView 多余的 handler.post
**文件**：`app/src/main/java/com/example/floating/ExecutionCardView.kt`

**问题**：
`updateStep` 和 `showCompletion` 都用 `handler.post { ... }` 包裹 TextView 赋值。调用方（`FloatingWindowManager.observeExecutionState`）已经在主线程的协程里运行，再 post 一次是冗余切换。

**修复**：去掉 `updateStep` 和 `showCompletion` 外层的 `handler.post`，保留内部的 `postDelayed(AUTO_HIDE_DELAY)` 用于延迟销毁。

---

### 10. ChatMessageAdapter DiffUtil 碰撞
**文件**：`app/src/main/java/com/example/ui/adapter/ChatMessageAdapter.kt`

**问题**：
`DiffUtil.areItemsTheSame` 用 `timestamp`（`System.currentTimeMillis()`）判等。同一毫秒内连发两条消息（UI 快速 append 时会发生）被 DiffUtil 认为是同一条，第二条不显示。

**修复**：给 `ChatMessageItem` 加 `id: Long`，由 companion object 里的 `AtomicLong` 发号，`areItemsTheSame` 改为比较 `id`。

---

### 11. 悬浮窗残留 Runnable 污染新任务
**文件**：`app/src/main/java/com/example/floating/FloatingWindowManager.kt`

**问题**：
`observeExecutionState` 在任务完成时 `handler.postDelayed({...}, 5500)` 安排清理：置 `executionCardView = null`、`showCircle()`、`ChatViewModel.resetState()`。如果 5.5 秒内用户又发起新任务：
- 新 Runnable 仍会在原时间触发
- `executionCardView = null` 会把新任务正在展示的卡片引用清掉
- `ChatViewModel.resetState()` 会把新任务的 `executionState` 清零，观察者看不到任何新状态

**修复**：
- 增加 `private var postTaskCleanup: Runnable? = null` 字段。
- 新任务开始（`isRunning && !isTaskRunning` 分支）时 `handler.removeCallbacks(postTaskCleanup)`。
- 任务完成时把 Runnable 存到字段里再 post。
