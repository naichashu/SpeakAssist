# 悬浮窗执行时偶发消失 Bug 分析与修复方案

- 报告时间：2026-05-07
- 影响版本：当前 main 分支（HEAD: 1d5e6b4）
- 实施分支：`fix/floating-window-disappear`（基于 main）
- Bug 描述（用户原话）：**"在执行的时候，悬浮窗有时候不会出现，特别是我在别人手机执行的时候，只要切换应用去执行，悬浮窗就没了，但是这个是偶然的，有时候又可以正常执行了"**
- 复现环境：用户自机 Android 14 极少触发；舍友手机 Android 15 频繁触发

---

## 一、结论先行

经过对 `ChatViewModel.executeTaskLoop`、`MyAccessibilityService` 截图相关 API、`FloatingWindowManager` 全部状态机、三个悬浮窗 View 类、`MainActivity` 生命周期回调的逐行核对，"悬浮窗偶发消失"是**5 个独立 bug 叠加**的复合表现，**不是单一原因，更不是 Android 15 系统差异**：

| 编号 | Bug | 严重度 | 真实存在 | 与现象的关系 |
|----|----|------|--------|-----------|
| B | `getForegroundPackageNameWithOverlayHidden` 必 hide，但 `isMyProjectApp` 分支跳过 restore | **P0** | 已验证 | 主因。每次主界面启动任务都触发，切应用瞬间至少缺席 1-3 秒 |
| A | `MainActivity.onDestroy → suspendFloatingOverlays` 把任务执行卡片也一起挂起 | **P0** | 已验证 | 在内存小 / ROM 激进的设备上频繁触发，舍友手机现象贴合此条 |
| C | `state.isCompleted && isTaskRunning` 在 A 触发后跳过 cleanup，状态泄漏 | P1 | 已验证 | A 的衍生症状。`_executionState` 卡在 isCompleted，下个任务覆盖才能恢复 |
| E | 手势期间 `reattachCancelChipAfterGesture` 单独把芯片挂回去，卡片仍缺席 | P2 | 已验证 | 偶发出现"光秃秃右上角取消芯片，下方无卡片"的怪异画面 |
| D | `hideAllOverlays` 把 `companionShouldShowCircle` 误清为 false | P3 | 已验证 | 极窄场景，`observeSettings` 大概率能自愈，影响微乎其微 |

**关于 Android 15 vs Android 14 的关系**：经查证 [Android 15 行为变化](https://developer.android.com/about/versions/15/behavior-changes-15)，与本 bug 直接相关的 API 行为变化只有"SYSTEM_ALERT_WINDOW + 前台服务"一条，且本项目未使用前台服务，**Android 15 不是直接成因**。"舍友手机更明显"的真实原因是其设备内存压力 / ROM 后台策略更激进，导致 Bug A 的 `MainActivity.onDestroy` 触发概率大幅上升。

**关于 Bug A 历史背景的重要发现**（与方案文档第一版的修正）：通过 `git log -S 'suspendFloatingOverlays'` 追到 commit `daab638` "修复悬浮窗残留与任务上下文串线"。当初引入 `suspendFloatingOverlays` 是为了修一个真实 bug：**用户从最近任务列表移除应用后，悬浮的圆形按钮还残留在屏幕上**。所以**直接删除 `MainActivity.onDestroy` 的调用会重新引入这个旧 bug**。本文档第一版给的方案 ①（删除调用）作废，改用方案 ②（细化 suspend 语义）。

## 二、Bug 详细分析

### 2.1 Bug B：截图前后 hide / restore 不对称（核心，必现）

#### 触发链路

```
ChatViewModel.executeTaskLoop step 0
    ↓
val currentApp = accessibilityService.getForegroundPackageNameWithOverlayHidden()
    ↓
内部：floatingWindowManager?.hideForScreenshot()       ← 无条件 hide
        delay(150)
        rootInActiveWindow.packageName
    ↓
回到 ChatViewModel：
val isMyProjectApp = currentApp == myProjectApp        ← 主界面启动时必然 true
    ↓
val screenShot = if (isMyProjectApp) {
    null                                                ← 跳过 takeScreenshotAfterForegroundCheck
} else {
    accessibilityService.takeScreenshotAfterForegroundCheck()
        ↑ 此函数内部才会调 floatingWindowManager?.restoreAfterScreenshot()
}
    ↓
❌ restoreAfterScreenshot 永远不被调
❌ executionCardView / executionCancelChipView 留在 markShowing=false 状态
    rootView 被 removeView 但实例和 LayoutParams 还在
```

#### 关键代码引用

- `ChatViewModel.kt:146-156`（hide / restore 不对称的入口）
  ```kotlin
  val currentApp = accessibilityService.getForegroundPackageNameWithOverlayHidden()
  val myProjectApp = getApplication<Application>().packageName
  val isMyProjectApp = currentApp == myProjectApp

  val screenShot = if (isMyProjectApp) {
      null
  } else {
      accessibilityService.takeScreenshotAfterForegroundCheck()
  }
  ```
- `MyAccessibilityService.kt:280-299`（hide 在前函数）
  ```kotlin
  suspend fun getForegroundPackageNameWithOverlayHidden(): String? {
      floatingWindowManager?.hideForScreenshot()    // ← 无条件
      delay(150)
      ...
  }
  ```
- `MyAccessibilityService.kt:305-313`（restore 在后函数）
  ```kotlin
  suspend fun takeScreenshotAfterForegroundCheck(): Bitmap? {
      val bitmap = withTimeoutOrNull(SCREENSHOT_TIMEOUT_MS) { ... }
      floatingWindowManager?.restoreAfterScreenshot()    // ← 只在调用方走"截图分支"时才执行
      return bitmap
  }
  ```

#### 用户视角的可见时间窗

```
T0      点击发送按钮
T0+ε    主界面上卡片闪现一瞬即被 hide（用户可能没注意）
T0+P    模型请求开始（无截图，纯文本上下文）
        卡片完全缺席
T0+P+R  模型返回 launch xxx
T0+P+R  ActionExecutor 启动目标 App
T0+P+R+settle  step 1 开始
        getForegroundPackageNameWithOverlayHidden 再次 hide（removeView 抛已脱离异常被吞）
        currentApp = 目标 App，isMyProjectApp = false
        takeScreenshotAfterForegroundCheck → restoreAfterScreenshot
        addView 把卡片重新挂到目标 App 上方  ✅
```

**空窗期长度 = 模型响应时间 + launch 执行 + settleDelay ≈ 1-3 秒**。这正是用户描述的"切换应用悬浮窗就没了"。

#### "偶然性"来源

不是 bug 本身偶然，而是**用户感知偶然**：
- 第 0 步模型回 `launch` → 第 1 步切到外部 App，restore 成功，几秒后卡片回来 → 用户觉得"正常"
- 第 0 步模型回 `finish`（任务在自己 App 内一步搞定，例如查看类任务） → 任务直接结束，卡片在整个执行期间从未真正出现过 → 用户觉得"完全没出现"
- 第 0 步模型回 `wait` / `tap` 在自己 App 内 → 第 1 步前台仍是 SpeakAssist，又走 `isMyProjectApp = true` 分支再次跳过 restore → **持续缺席直到模型让 launch**

### 2.2 Bug A：`MainActivity.onDestroy → suspendFloatingOverlays` 把执行卡片也一起挂起（次要但严重）

#### 历史背景（新发现）

`commit daab638` "修复悬浮窗残留与任务上下文串线" 引入了这套 suspend 机制，目的是修复"用户在最近任务列表里划掉 SpeakAssist，但悬浮的圆形按钮还在屏幕上"。这个修复**思路是对的**，但**粒度过粗**：把所有悬浮窗都挂起了，包括正在显示任务进度的执行卡片和取消芯片。

#### 触发链路

```
某些场景触发 MainActivity.onDestroy：
- 用户从最近任务划掉 SpeakAssist 卡片
- 系统因低内存回收 Activity（别人手机高发）
- 开发者选项"不保留活动"开启
    ↓
MainActivity.onDestroy() → MyAccessibilityService.suspendFloatingOverlays()
    ↓
FloatingWindowManager.suspendOverlays()
    overlaysSuspended = true
    handler.post {
        stopWakeWordListening()
        hideAllOverlays()    ← ❌ 过度：连任务卡片都销毁了
    }
```

#### 锁死后无法恢复的路径

```kotlin
// 任务执行中 ChatViewModel 还在调用：
suspend fun restoreAfterScreenshot() {
    ...
    if (overlaysSuspended || !isTaskRunning) return@withContext   // ← 双重挡住
    ...
}
private fun showExecutionCard(title: String) {
    if (overlaysSuspended) return                                  // ← 挡住
    ...
}
```

唯一解锁入口是 `MainActivity.onCreate → resumeFloatingOverlays`。**用户不回主界面，任务卡片永远瘫痪**。

#### 设计层面的权衡

需要区分两类悬浮窗的语义：

| 悬浮窗 | 性质 | Activity 销毁时应该 |
|----|----|----|
| `CircleFloatingView`（圆球） | 空闲入口，跟"用户感知 App 还活着"绑定 | **隐藏**（修 daab638 的旧 bug） |
| `ExecutionCardView`（任务卡片） | 任务实时反馈，跟"任务是否在跑"绑定 | **保留**（让任务 finish 自己来清理） |
| `ExecutionCancelChipView`（取消芯片） | 跟卡片同生命周期 | **保留** |
| 唤醒词监听 | 跟"想被唤醒"绑定，不跟 Activity 绑定 | 由 `voiceWakeEnabled` 控制，不动 |

`daab638` 的 patch 把这三件事用同一个 flag 控制，所以才会出现"为了修圆球残留，把任务卡片也一起干掉"的副作用。

#### 为什么"在别人手机更明显"

| 因素 | 你的手机 | 舍友手机 |
|----|--------|--------|
| 物理内存 | 大 | 小 |
| 后台白名单 | SpeakAssist 已加 | 未加 |
| ROM 厂商策略 | AOSP / 类原生 | MIUI / ColorOS / OriginOS 等 |
| onDestroy 触发频率 | 罕见 | 频繁 |

→ 你几乎遇不到，舍友每次切应用都可能触发。

### 2.3 Bug C：cleanup 分支在 A 触发后被跳过（A 的衍生症状）

#### 触发链路

```
Bug A 已触发，hideAllOverlays 把 isTaskRunning = false
    ↓
ChatViewModel 继续执行任务（companion StateFlow 不受影响）
    ↓
任务结束：_executionState.value = ...copy(isCompleted = true, isRunning = false, ...)
    ↓
FloatingWindowManager.observeExecutionState 收到
    ↓
when {
    state.isCompleted && isTaskRunning -> { ... }    ← isTaskRunning 已是 false
}
    ↓
❌ 整个清理分支跳过：
    × hideExecutionCancelChip 不调
    × showCompletion 不显示
    × cleanup runnable 不排程
    × ChatViewModel.resetState() 不调
    ↓
_executionState 卡在 isCompleted=true 状态，直到下个任务覆盖
```

#### 后果

- 短期：用户重新打开主界面后能正常发新任务（覆盖 _executionState），所以"看起来好了"
- 长期：状态泄漏不引发功能性卡死，但 `MainActivity.sendMessage` 的并发护栏（`MainActivity.kt:417`）依赖 `executionState.value.isRunning`，如果未来加更多基于 `isCompleted` 的逻辑会踩坑

### 2.4 Bug E：手势期间芯片单独 reattach 造成画面错乱（B 的衍生）

#### 触发链路

```
Bug B 触发：step 0 期间 card 和 chip 都被 hide，markShowing=false
    ↓
模型恰好在 step 0 给出非 launch 的手势动作（tap / swipe / longpress / doubletap）
    ↓
ActionExecutor 执行手势 → MyAccessibilityService.dispatchGestureAwaiting
    ↓
detachCancelChipForGesture()    [FloatingWindowManager.kt:380]
    ↓ 这里只动 chip，不动 card
    ↓ 但 chip 已经被 hide 了，markShowing=false
    ↓ rootView?.let { windowManager.removeView(view) } 抛异常被 catch
    ↓
gesture 执行
    ↓
reattachCancelChipAfterGesture()    [FloatingWindowManager.kt:395]
    ↓
检查：if (!chip.isShowing()) addView(...)
    ↓
✅ 芯片被单独挂回去
❌ 但 card 还是缺席
    ↓
用户看到："右上角光秃秃一个'取消执行'芯片，下面什么都没有"
```

#### 后果

视觉怪异，但不影响功能。`reattachCancelChipAfterGesture` 把 chip 单独修好之后，下一次 step 进来时 `restoreAfterScreenshot` 才会同时把 card 也挂上。中间这段时间存在"只见芯片不见卡片"的不一致状态。

### 2.5 Bug D：`companionShouldShowCircle` 被错清（影响微乎其微）

#### 触发链路

```
hideAllOverlays() 被 suspendOverlays 或 destroy 调用
    ↓
companionShouldShowCircle = false       [FloatingWindowManager.kt:202]
companionWasTaskRunning = false
```

#### 语义错误

`companionShouldShowCircle` 这个跨实例标志的语义是"AccessibilityService 重建后，新实例据此恢复 circle"。`destroy` 时清理是合理的，但 `suspendOverlays` 是**临时**挂起，清掉这个标志破坏了"稍后能恢复"的契约。

#### 自愈机制掩盖了 bug

实际场景中 `observeSettings` 的 `floatingWindowEnabled.collect` 第一次会发出当前值，触发 `if (enabled && !isTaskRunning) showCircle()` —— `showCircle` 内部会重置 `companionShouldShowCircle = true`，所以**绝大多数路径能自愈**。仅"suspend 期间无障碍服务被重启 + 用户没回主界面"这种很窄的场景才会暴露。

## 三、修复方案

修复按优先级分三批，可独立实施。**P0 两条必做**，能消除 90%+ 的现象；P1/P2/P3 是补强。

### 3.1 修复 Bug B（P0，最高优先级）— 推荐方案 ②

#### 方案对比

| 方案 | 改动量 | 优点 | 缺点 |
|---|---|---|---|
| ① 在 `isMyProjectApp` 分支补一次 `restoreAfterScreenshot` | 极小 | 改动最少 | 治标不治本，hide / restore 仍分散在两个函数；将来加新分支还会忘 |
| **② 把 hide / restore 配对成一个 try-finally suspend 块（推荐）** | 中等 | 从根本上保证对称，编译期防忘；语义清晰 | 需要重构 `MyAccessibilityService` 的两个函数 |
| ③ 把 `isMyProjectApp` 检测前置到 hide 之前（用 `_currentApp` StateFlow） | 较大 | 不需要 hide 就能拿到包名 | StateFlow 可能滞后于真实前台，引入新的偏差 |

#### 推荐实施（方案 ②）

**修改点 1：`MyAccessibilityService.kt`** — 新增一个把 hide → 操作 → restore 包成一个块的高阶 suspend 函数

```kotlin
/**
 * 用于"获取前台 App 包名 + 可选截图"的统一入口。
 * 保证 overlay hide 一定有配对的 restore，无论 block 是否抛异常或提前 return。
 *
 * @param block 在 overlay 已隐藏的状态下执行，可访问 rootInActiveWindow / takeScreenshot
 */
suspend fun <T> withOverlayHidden(block: suspend () -> T): T {
    floatingWindowManager?.hideForScreenshot()
    return try {
        delay(150)              // 等 overlay 完全从窗口树中消失
        block()
    } finally {
        floatingWindowManager?.restoreAfterScreenshot()
    }
}
```

**修改点 2：`MyAccessibilityService.kt`** — 删除原来的两个函数 `getForegroundPackageNameWithOverlayHidden()` 和 `takeScreenshotAfterForegroundCheck()`，避免调用方再用错。

**修改点 3：`ChatViewModel.kt:146-156`** — 调用方改写

```kotlin
val (currentApp, screenShot) = accessibilityService.withOverlayHidden {
    val pkg = accessibilityService.rootInActiveWindow?.packageName?.toString()
        ?: accessibilityService.currentApp.value
    val bmp = if (pkg == myProjectApp) {
        null
    } else {
        suspendCancellableCoroutine<Bitmap?> { cont ->
            accessibilityService.getScreenshot { cont.resume(it) }
        }
    }
    pkg to bmp
}
```

> 注意：`rootInActiveWindow` 的 `AccessibilityNodeInfo` 必须 `recycle()` 或 `close()`（项目已有此惯例，详见 `MyAccessibilityService.kt:60-69` 的写法），实际实现时要按项目惯例 close。

#### 验证方法

- 修改后从主界面发"打开微信"
- 应观察到：切到微信瞬间执行卡片**立即**出现在微信上方，无 1-3 秒空白
- 模型回 `finish` 的查看型任务（例如"看一下当前页面"），执行卡片应在主界面就完整显示出 5.5 秒完成态再消失

### 3.2 修复 Bug A（P0）— 细化 suspend 语义

**注意：本节是相对方案文档第一版的重大修正**。第一版推荐"直接删除 `MainActivity.onDestroy` 的调用"会重新引入 daab638 当初修复的"圆球残留" bug，**作废**。

#### 方案对比

| 方案 | 改动量 | 优点 | 缺点 |
|---|---|---|---|
| ❌ 删除 `MainActivity.onDestroy` 中的调用 | 极小 | 简单 | **重新引入 daab638 修复的圆球残留 bug** |
| **✅ 推荐：suspend 只销毁圆球 + 唤醒词，不动任务卡片** | 中 | 保留 daab638 修复，同时让任务卡片不受影响 | 改动 FloatingWindowManager 几个状态检查点 |
| 把 `overlaysSuspended` 标志拆成两个（circleSuspended / cardSuspended） | 大 | 更清晰 | 状态机更复杂，没必要 |

#### 推荐实施

**核心思路**：`suspendOverlays` 的语义应该是"用户感知 App 关了"，所以只该影响"App 入口性质"的圆球，**不应该影响任务执行进度反馈**（卡片和取消芯片）。任务执行的生命周期由 `isTaskRunning` 控制，跟 Activity 解耦。

**修改点 1：`FloatingWindowManager.suspendOverlays`** — 只销毁 circle

```kotlin
fun suspendOverlays() {
    overlaysSuspended = true
    handler.post {
        stopWakeWordListening()
        // 只销毁圆球（修 daab638 的"残留圆球"bug 的本意）
        // 不要碰 executionCardView / executionCancelChipView：
        // 任务执行中即使 Activity 销毁，用户仍需看到进度
        circleView?.destroy()
        circleView = null
    }
}
```

**修改点 2：`showExecutionCard` / `showExecutionCancelChip`** — 删除 overlaysSuspended 检查

```kotlin
private fun showExecutionCard(title: String) {
    // 删除：if (overlaysSuspended) return
    handler.post { ... }
}
```

**修改点 3：`restoreAfterScreenshot` / `reattachCancelChipAfterGesture`** — 只检查 isTaskRunning

```kotlin
suspend fun restoreAfterScreenshot() {
    withContext(Dispatchers.Main.immediate) {
        // 删除 overlaysSuspended 检查；只看任务是否在跑
        if (!isTaskRunning) return@withContext
        ...
    }
}
```

**修改点 4：`showCircle`** — 保留 overlaysSuspended 检查（圆球受 suspend 影响，正确）

#### 验证方法

- 修改后从主界面发任务 → 切到目标 App → 在最近任务里手动划掉 SpeakAssist
- 应观察到：
  1. **圆球消失了**（保留 daab638 的修复）
  2. **执行卡片继续在目标 App 上显示进度，任务正常进行到结束**（修复 Bug A）
- 开发者选项打开"不保留活动"，重复测试，结果应一致

### 3.3 修复 Bug C（P1）— 解耦 cleanup 与 isTaskRunning

A 修好后 C 就基本不会触发，但仍建议加固以防类似 bug 再现。

**修改点：`FloatingWindowManager.kt:490-512`**

### 3.4 修复 Bug E（P2）— reattach 时同步检查 card

**修改点：`FloatingWindowManager.kt:395-409`**

```kotlin
suspend fun reattachCancelChipAfterGesture() {
    withContext(Dispatchers.Main.immediate) {
        if (overlaysSuspended || !isTaskRunning) return@withContext
        // 同时检查并恢复 card，避免出现"光秃秃芯片无卡片"的不一致画面
        executionCardView?.takeIf { !it.isShowing() }?.let { card ->
            card.rootView?.let { view ->
                card.currentLayoutParams?.let { params ->
                    try {
                        windowManager.addView(view, params)
                        card.markShowing(true)
                    } catch (e: Exception) {
                        Log.w(TAG, "手势后恢复卡片失败", e)
                    }
                }
            }
        }
        // 原有 chip 恢复逻辑保持不变
        val chip = executionCancelChipView ?: return@withContext
        ...
    }
}
```

#### 验证方法

## 四、可选优化（不属于 bug 修复，但解决长期隐患）

### 4.1 把 `TYPE_APPLICATION_OVERLAY` 换成 `TYPE_ACCESSIBILITY_OVERLAY`

#### 现状

三个悬浮窗 View 都用 `TYPE_APPLICATION_OVERLAY`：
- `CircleFloatingView.kt:108-113`
- `ExecutionCardView.kt:61-66`
- `ExecutionCancelChipView.kt:49-54`

#### 推荐

由于 SpeakAssist 是 AccessibilityService，按 [Android 官方推荐](https://developer.android.com/reference/android/accessibilityservice/AccessibilityService)应使用 `TYPE_ACCESSIBILITY_OVERLAY`：

| 维度 | TYPE_APPLICATION_OVERLAY | TYPE_ACCESSIBILITY_OVERLAY |
|---|---|---|
| 权限要求 | SYSTEM_ALERT_WINDOW | BIND_ACCESSIBILITY_SERVICE |
| 信任级别 | 不受信任 | 受信任的系统窗口 |
| 在系统设置/权限弹窗上的可见性 | 被遮挡 | 仍可见 |
| Android 12+ touch-blocking 限制 | 受影响 | 不受影响 |
| Android 15 SYSTEM_ALERT_WINDOW + fgs 限制 | 间接受影响 | 不受影响 |
| 需要用户授权 | 是（需引导用户去设置开） | 否（绑定服务时自动获得） |

#### 改动

- 三个 view 的 overlayType 一行替换
- AndroidManifest 中的 `android.permission.SYSTEM_ALERT_WINDOW` **可以删除**（如果完全替换）
- 主界面的"悬浮窗权限申请"流程可以简化甚至移除

#### 风险

- 需在两台手机（你和舍友）实测，部分国产 ROM 对 `TYPE_ACCESSIBILITY_OVERLAY` 的实现可能有差异
- 切换后悬浮窗在系统级 dialog 上也会显示，需评估是否符合预期

#### 建议

**先修 Bug B + A，确认现象消失。如果还有边角问题再考虑这条**。这条是优化，不是 bug 修复，不在主修复路径上。

## 六、风险评估

| 修复 | 引入风险 | 缓解措施 |
|---|---|---|
| Bug B 修复 | `withOverlayHidden` 块内抛异常时 finally 仍执行 restore，可能在异常路径下尝试恢复"已不存在"的 view（addView 抛异常） | restoreAfterScreenshot 内部已有 try-catch 保护，不会扩散 |
| Bug A 修复 | `suspendOverlays` 只销毁 circle 后，任务卡片在 Activity 死亡期间仍存在；如果任务在此期间因 lifecycleScope 取消，卡片会 stuck | `lifecycleScope` cancel 触发 `executeTaskLoop` 的 catch 块调 `finishTask(isCancelled=true)`，会发出 `isCompleted=true` → cleanup runnable 销毁卡片。链路可走通 |
| Bug C 修复 | `state.isCompleted` 不再 && isTaskRunning，可能在重复 isCompleted 事件上重复执行 | StateFlow 同值不发射，只有 `_executionState.value` 真正变化才触发 collect，无重入风险 |
| Bug E 修复 | reattach 时同时操作 card 和 chip，双倍 addView 失败的可能性 | 各自 try-catch，独立恢复 |
| 4.1 OVERLAY 类型替换 | 国产 ROM 兼容性差异 | 分批小规模灰度验证，不在主修复路径 |
