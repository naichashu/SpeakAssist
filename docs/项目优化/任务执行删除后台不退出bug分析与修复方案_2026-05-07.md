# 任务执行过程中删除后台任务不退出 bug 分析与修复方案

- 报告时间：2026-05-07
- 影响版本：当前 main 分支（HEAD: 1d5e6b4）
- Bug 描述（用户原话）：**"任务执行过程中，如果立马删除后台，任务有概率还在执行"**

---

## 一、结论先行

经过对 `ChatViewModel`、`MyAccessibilityService`、`FloatingWindowManager`、`ActionExecutor`、`ModelClient` 的源码与 Kotlin 协程取消语义的逐行核对，"任务还在执行"是**两个独立 bug 叠加**的复合表现，**不是单一原因**：

| 编号 | Bug | 严重度 | 真实存在 | 用户感知 |
|----|-----|------|--------|--------|
| A | `finishTask` 在已 cancel 协程里先调 suspend `db.update`，导致 `_executionState` 不更新 | **P0** | 已验证 | UI 卡死、看起来"任务还在执行"，重启 App 也发不了新任务 |
| B | `FloatingWindowManager.onVoiceResult` 用 service-scope 启动 `executeTaskLoop`，划应用不会取消 | **P0** | 已验证 | 任务真的还在跑、屏幕上动作真实发生 |
| C | `MainActivity.onDestroy()` 没显式调 `ChatViewModel.requestCancel()`，依赖 `lifecycleScope` 协作取消 | P1 | 已验证 | 加剧 A 的概率窗口 |
| D | `ActionExecutor` 11 处 `catch (e: Exception)` 吞 `CancellationException` | P2 | 已验证 | 仅污染日志、误导 UI step 计数；不会让 loop "再走一步" |

**与第一轮分析的修正**：第一轮我把 D 列为"主因之一"。重新核对后发现 D 被吞之后立即进入 retry 流程，retry 内只有 `delay(1000)` 一个 cancellation point，会**立即**抛出 `CancellationException` 跳出循环——所以 D **不会让任务真的多走一步动作**，最坏只让退出延迟 1 秒。真正的"任务还在执行"是 A 和 B。

---

## 二、Bug 详细分析

### 2.1 Bug A：`finishTask` 状态更新被 cancel 截断（核心）

#### 触发链路

```
用户划掉应用 (从最近任务列表)
    ↓
MainActivity.onDestroy()
    ↓
lifecycleScope.cancel()    （Lifecycle.State.DESTROYED 自动触发）
    ↓
executeTaskLoop 的当前 suspend 点抛 CancellationException
    ↓
catch (e: kotlinx.coroutines.CancellationException) 块执行
    ↓
finishTask(sessionId, false, ..., isCancelled = true)    [ChatViewModel.kt:361]
    ↓
finishTask 内部第一行 suspend 调用：
    db.taskSessionDao().updateStatus(sessionId, dbStatus)    [ChatViewModel.kt:410]
    ↓
【关键】此时协程 Job.isCancelled == true，
suspend 调用立即抛 CancellationException
（Kotlin 协程 "Prompt cancellation guarantee"）
    ↓
后面的代码不再执行：
    × messageContext.clear()
    × _executionState.value = ...copy(isCompleted = true, ...)    [ChatViewModel.kt:412]
    × return TaskResult(...)
```

#### 后果

1. `_executionState` 是 **companion object 单例**（`ChatViewModel.kt:66-67`），进程级共享。
2. `MyAccessibilityService` 是系统 bound service，划应用**不会**让进程被 kill —— companion 状态保留。
3. `FloatingWindowManager.observeExecutionState()` 一直 collect `_executionState`（`FloatingWindowManager.kt:475-516`），但永远收不到 `isCompleted = true`：
   - `isTaskRunning` 永远停在 `true`
   - `executionCardView` 永远不被 `showCompletion` + 5.5 秒后清理
   - `executionCancelChipView` 永远不被 `hideExecutionCancelChip`
   - `circleView` 永远不被恢复
4. 用户重启 App，`MainActivity.sendMessage()` 的并发护栏（`MainActivity.kt:417`）：
   ```kotlin
   if (ChatViewModel.executionState.value.isRunning) {
       Toast.makeText(this, R.string.task_already_running, ...)
       return
   }
   ```
   **永远命中**，用户发不了新任务。

#### 关键证据（Kotlin 协程语义）

> All the suspending functions in `kotlinx.coroutines` are cancellable. They check for cancellation of coroutine and throw CancellationException when cancelled.
> —— [Kotlin 官方文档](https://kotlinlang.org/docs/cancellation-and-timeouts.html)

Room 的 suspend DAO、`kotlinx.coroutines.delay`、`withContext` 全部满足上述语义。所以 `db.taskSessionDao().updateStatus(...)` 在 cancel 状态下**保证立即抛 CancellationException**。

### 2.2 Bug B：悬浮窗启动的任务在 service-scope，划应用完全不停（核心）

#### 路径

```kotlin
// FloatingWindowManager.kt:43
private val scope = CoroutineScope(SupervisorJob() + Dispatchers.Main)

// FloatingWindowManager.kt:411-437
private fun onVoiceResult(text: String) {
    ...
    scope.launch {                                          // <- service-owned
        val viewModel = ChatViewModel(service.application)  // <- service.application
        val result = viewModel.executeTaskLoop(text, ...)
    }
}
```

`scope.cancel()` 只在 `FloatingWindowManager.destroy()` 调用（`FloatingWindowManager.kt:181`），而 `destroy()` 只在 `MyAccessibilityService.onDestroy()` 触发（`MyAccessibilityService.kt:109-114`）。

`MyAccessibilityService.onDestroy()` **仅在用户在系统设置里关闭无障碍服务时调用**。划应用：
- Activity onDestroy（lifecycleScope cancel）：会触发
- AccessibilityService onDestroy（FloatingWindowManager.scope cancel）：**不会**触发
- 进程被 kill：bound service 保活，**不会**触发

**所以从悬浮窗启动的任务，划应用后 100% 继续执行**，且会真实在屏幕上做动作。

#### 触发条件

- 用户启用悬浮窗 + 语音唤醒
- 用唤醒词触发悬浮窗执行 → `onVoiceResult` 启动任务
- 用户回到桌面或最近任务划掉应用

可能用户不知道的是：**只要悬浮窗启动了任务，划应用根本停不下来**，必须点取消芯片。

### 2.3 Bug C：`MainActivity.onDestroy()` 缺显式取消（次要）

```kotlin
// MainActivity.kt:745-751
override fun onDestroy() {
    super.onDestroy()
    speechManager.destroy()
    noiseLevelJob?.cancel()
    voiceBackgroundAnimator?.cancel()
    MyAccessibilityService.suspendFloatingOverlays()
    // 没有 ChatViewModel.requestCancel()
}
```

依赖 `lifecycleScope` 自动 cancel，问题是：
1. `_cancelRequested` 始终 false → loop 内 `if (_cancelRequested.value)` 主动检查（`ChatViewModel.kt:136-140`）形同虚设，无法主动 `modelClient?.cancelCurrentRequest()`。
2. HTTP 请求要等 OkHttp 收到 `currentCall?.cancel()`（通过 `continuation.invokeOnCancellation`，`ModelClient.kt:165-168`）才会断开。理论上 `lifecycleScope.cancel()` 也会触发同一条 invokeOnCancellation，但有时间窗口。
3. 加剧 Bug A 发生概率：取消信号到来的"瞬间状态"更不可控。

### 2.4 Bug D：`ActionExecutor` 吞 `CancellationException`（弱影响）

11 处 `catch (e: Exception)`：

```
ActionExecutor.kt:232  // execute - JSON 修复路径
ActionExecutor.kt:245  // execute - lenient 解析
ActionExecutor.kt:282  // execute - 顶层
ActionExecutor.kt:316  // extractJsonFromText
ActionExecutor.kt:384  // executeAction
ActionExecutor.kt:432  // launchApp
ActionExecutor.kt:662  // swipe
ActionExecutor.kt:691  // back
ActionExecutor.kt:720  // home
ActionExecutor.kt:789  // longPress
ActionExecutor.kt:858  // doubleTap
```

`kotlinx.coroutines.CancellationException` 继承自 `IllegalStateException` → `RuntimeException` → `Exception`，所以全部会被吞。

**但实际影响有限**：吞掉后返回 `ActionResult(success = false, ...)`，loop 进入 retry 分支：

```kotlin
// ChatViewModel.kt:278-342
if (!result.success) {
    ...                  // 同步代码：messageContext 操作、retryCount++
    delay(1000)          // <- suspend，在已 cancel 协程里立即抛 CancellationException
    continue
}
```

`delay(1000)` 立即抛，loop 退出。所以 D **最多让退出延迟同步代码的几百微秒**，不会"多执行一个动作"。

**主要危害是污染日志和误导**：`ActionResult.message = "执行动作失败: tap - StandaloneCoroutine was cancelled"` 这种错误会被 `executeTaskLoop` 写进 `TaskStep` 数据库表（`ChatViewModel.kt:262-270`），让历史记录看起来像是"动作执行失败"，掩盖真正的取消语义。

## 四、修复方案

### 4.1 修复 Bug A：`finishTask` 用 `NonCancellable` 保护关键状态

**最小补丁**（推荐）：

`app/src/main/java/com/example/ui/viewmodel/ChatViewModel.kt:399-420`

```kotlin
import kotlinx.coroutines.NonCancellable
import kotlinx.coroutines.withContext

private suspend fun finishTask(
    sessionId: Long,
    success: Boolean,
    message: String,
    isCancelled: Boolean = false
): TaskResult {
    val dbStatus = when {
        isCancelled -> "cancelled"
        success -> "success"
        else -> "fail"
    }

    // 关键：UI 状态先同步更新（不依赖 suspend），保证不被取消信号截断
    messageContext.clear()
    _executionState.value = _executionState.value.copy(
        isRunning = false,
        isCompleted = true,
        isSuccess = success,
        isCancelled = isCancelled,
        resultMessage = message
    )

    // 数据库写入用 NonCancellable 保证落盘（即使外层 Job 已 cancel）
    withContext(NonCancellable) {
        db.taskSessionDao().updateStatus(sessionId, dbStatus)
    }

    return TaskResult(success, message)
}
```

**要点**：

- **顺序很关键**：`_executionState.value = ...` 是同步赋值，必须放在任何 suspend 调用之前。
- `withContext(NonCancellable)` 保证 db 写入不被 cancel 中断；即使 cancel 也能落盘 status。
- `messageContext.clear()` 也是同步操作，放前面。
- `return` 在 cancel 协程里仍会抛 `CancellationException`（因为整个 finishTask 还在 cancelled 协程上下文内），但已经不影响 UI 状态。

### 4.2 修复 Bug B：service-scope 启动的任务挂到任务自身的 Job 上，并在划应用时统一取消

由于 service-scope 任务"能在后台跑"是设计意图（用户用悬浮窗就是为了不开 App），**不能简单粗暴地让划应用就停**。但需要给用户一个明确的、和 MainActivity 启动一致的取消通道。

**推荐方案**：让 `ChatViewModel.requestCancel()` 同时取消 HTTP 请求 + 设置 `_cancelRequested`，并在以下三处都调它：

1. 取消芯片点击（已有，`FloatingWindowManager.kt:306`）。
2. **新增**：`MainActivity.onDestroy()`（修复 Bug C，见 4.3）。
3. **新增**：在 `FloatingWindowManager.onVoiceResult` 里，把 launch 的 Job 保存下来，方便外部 cancel：

```kotlin
// FloatingWindowManager.kt
private var currentTaskJob: Job? = null

private fun onVoiceResult(text: String) {
    ...
    currentTaskJob?.cancel()  // 防御：旧任务先取消
    currentTaskJob = scope.launch {
        try {
            val viewModel = ChatViewModel(service.application)
            viewModel.executeTaskLoop(text, "autoglm-phone")
        } catch (e: CancellationException) {
            ChatViewModel.resetState()
            throw e
        } catch (e: Exception) {
            Log.e(TAG, "任务执行失败", e)
            ChatViewModel.resetState()
        } finally {
            currentTaskJob = null
        }
    }
}
```

并加一个产品决策开关（写入 SettingsPrefs）：

> **设置 → 任务执行 → 划掉应用时是否中断后台任务**（默认开启）

如果开启：在 `MainActivity.onDestroy()` 里调 `FloatingWindowManager.cancelCurrentTask()` 或直接 `ChatViewModel.requestCancel()`。
如果关闭：保持当前行为（任务后台继续）。

**最简方案（不加开关）**：直接在 `MainActivity.onDestroy()` 里调 `ChatViewModel.requestCancel()`——这同时盖住了 Bug C，并让 service-scope 启动的任务也能在划应用时收到取消信号。

```kotlin
// MainActivity.kt:745
override fun onDestroy() {
    super.onDestroy()
    // 用户划应用时主动取消正在跑的任务（包括悬浮窗启动的 service-scope 任务）。
    // _cancelRequested 是 companion object，所有 ChatViewModel 实例共享。
    if (ChatViewModel.executionState.value.isRunning && !isChangingConfigurations) {
        ChatViewModel.requestCancel()
    }
    speechManager.destroy()
    noiseLevelJob?.cancel()
    voiceBackgroundAnimator?.cancel()
    MyAccessibilityService.suspendFloatingOverlays()
}
```

**为什么要 `!isChangingConfigurations`**：横竖屏切换会触发 onDestroy，但用户没有主动想取消任务。

### 4.4 修复 Bug D：`ActionExecutor` 重抛 `CancellationException`

**最小补丁**：在 11 处 `catch (e: Exception)` 顶部都加一行重抛。

示例（`ActionExecutor.kt:282` 顶层 execute）：

```kotlin
suspend fun execute(actionJson: String, screenWidth: Int, screenHeight: Int): ActionResult {
    return try {
        ...
    } catch (e: kotlinx.coroutines.CancellationException) {
        throw e   // 必须重抛，让协程取消语义透传
    } catch (e: Exception) {
        Log.e(TAG, "解析动作失败", e)
        ActionResult(success = false, message = "解析动作失败: ${e.message}")
    }
}
```

需要修改的所有点（11 处）：

| 文件:行 | 函数 | 改动 |
|--------|------|-----|
| `ActionExecutor.kt:232` | `execute` 内部 fixedJson 路径 | 加 `catch (CancellationException) { throw it }` |
| `ActionExecutor.kt:245` | `execute` lenient 解析 fallback | 同上 |
| `ActionExecutor.kt:282` | `execute` 顶层 | 同上 |
| `ActionExecutor.kt:316` | `extractJsonFromText` 候选筛选 | 同上 |
| `ActionExecutor.kt:384` | `executeAction` | 同上 |
| `ActionExecutor.kt:432` | `launchApp` | 同上 |
| `ActionExecutor.kt:662` | `swipe` | 同上 |
| `ActionExecutor.kt:691` | `back` | 同上 |
| `ActionExecutor.kt:720` | `home` | 同上 |
| `ActionExecutor.kt:789` | `longPress` | 同上 |
| `ActionExecutor.kt:858` | `doubleTap` | 同上 |

**更优替代**：直接收紧 catch 类型，比如 `catch (e: RuntimeException)` + `catch (e: IOException)` 分开处理，但改动较大，建议先用最小补丁。
